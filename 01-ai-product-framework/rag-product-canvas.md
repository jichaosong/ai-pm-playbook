# RAG 产品画布

> **RAG 产品画布（RAG Product Canvas）** 是面向检索增强生成（Retrieval-Augmented Generation）产品的分析框架。RAG 是当前 AI 产品落地最主流的技术范式之一，通过将外部知识库的检索结果注入 LLM 上下文，有效解决幻觉、知识时效性、私有数据访问等核心问题。

---

## 一、RAG 产品架构全景

```
用户问题
    │
    ▼
┌─────────────┐     ┌─────────────┐
│  查询理解    │────→│  检索策略    │
│  (Query     │     │  (Retrieval) │
│   Rewrite)  │     │              │
└─────────────┘     └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  检索结果     │
                    │  (Chunks)    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  上下文组装   │←── 上下文窗口策略
                    │  (Context    │
                    │   Assembly)  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  LLM 生成    │
                    │  (Generation)│
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  后处理 &     │
                    │  评估 & 反馈  │
                    └──────────────┘
```

### 模块 1：场景定义

**核心问题：RAG 要解决什么业务场景？为什么需要 RAG？**

#### 1.1 RAG 适用场景判断

| 适用场景 ✅ | 不适用场景 ❌ |
|------------|-------------|
| 需要引用特定知识库回答 | 纯创意写作（不需要外部知识） |
| 知识频繁更新（如新闻、政策） | 实时性要求 < 100ms 的查询 |
| 私有/内部知识不可见 LLM | 只需要常识回答 |
| 需要提供引用来源 | 查询范围极窄且固定 |
| 需要降低幻觉风险 | 不需要事实性验证 |

#### 1.2 场景分类

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| **企业知识问答** | 基于内部文档回答问题 | HR 政策查询、产品手册问答 |
| **客服 FAQ** | 基于产品知识库解答用户问题 | 电商退换货政策、技术支持 |
| **文档分析** | 基于文档内容进行分析总结 | 财报分析、法律合同审查 |
| **编程问答** | 基于代码库/文档回答技术问题 | 框架 API 查询、Bug 修复建议 |
| **教育辅导** | 基于教材内容提供学习支持 | 解题辅导、知识点讲解 |
| **数据分析** | 基于数据表格/报告回答问题 | 销售数据查询、运营指标解读 |

**设计原则：**
- 确认场景是否真的需要检索——如果 LLM 自身已能很好地回答，RAG 可能过度工程化
- 评估知识库的规模和更新频率——小规模静态知识库可能直接用 embedding 即可

### 模块 2：知识库设计

**核心问题：知识从哪里来？如何组织才能使检索高效？**

#### 2.1 知识源评估

| 维度 | 说明 | 评估标准 |
|------|------|----------|
| **数据来源** | 文档、数据库、网页、API 等 | 质量、权威性、完整性 |
| **格式多样性** | PDF、Markdown、HTML、Word、CSV | 解析难度（不同格式解析差异大） |
| **数据量** | 文档数量、总 Token 数 | 影响检索策略和成本 |
| **更新频率** | 知识的新鲜度要求 | 影响索引更新策略 |
| **权限控制** | 哪些用户能看到哪些知识 | 影响检索的权限过滤 |

#### 2.2 知识库架构

```
文档摄取 pipeline：
源文档 → 格式解析 → 清洗 → 分块 → Embedding → 存入向量库
                                          ↓
                                    元数据存储 (MD)
```

| 组件 | 选项 | 选型考虑 |
|------|------|----------|
| **向量数据库** | Pinecone / Weaviate / Milvus / Qdrant / Chroma | 规模、延迟、托管 vs 自建 |
| **Embedding 模型** | text-embedding-3-small / bge-large / M3E | 维度、性能、语言支持 |
| **混合检索** | 向量 + BM25 / TF-IDF | 提高检索精确度 |
| **元数据过滤** | 文档类型、时间范围、部门、标签 | 权限控制和范围限定 |

#### 2.3 知识库更新策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **全量重建** | 定期重新处理全部文档 | 小规模知识库（<1M chunks） |
| **增量更新** | 只更新变化的部分 | 大规模知识库、高频更新 |
| **实时索引** | 文档更新后立即索引 | 实时性要求高的场景 |
| **定时批量** | 每天/每周定时更新 | 知识更新有固定节奏 |

### 模块 3：分块策略

**核心问题：如何将文档切分为最适合检索的片段？**

这是 RAG 产品最容易被忽视但影响最大的环节之一。

#### 3.1 分块参数

| 参数 | 说明 | 推荐值 | 对效果的影响 |
|------|------|--------|-------------|
| **块大小 (Chunk Size)** | 每个块的 Token 数 | 256-1024 | 太小则信息不完整，太大则噪声多 |
| **块重叠 (Chunk Overlap)** | 相邻块的重叠 Token 数 | 10%-20% | 防止关键信息被截断 |
| **分块策略** | 固定长度 vs 语义边界 | 语义边界优先 | 语义完整块效果显著优于随机切分 |

#### 3.2 分块策略对比

| 策略 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **固定长度** | 按 Token 数均匀切分 | 实现简单、速度快 | 可能打断语义完整的段落 |
| **段落分块** | 按自然段落（换行）切分 | 语义完整 | 段落长度差异大 |
| **递归分块** | 按层级分隔符递归切分（段落→句子→短语） | 兼顾语义和大小控制 | 实现复杂 |
| **语义分块** | 使用 Embedding 检测主题变化点 | 语义最完整 | 计算成本高 |
| **滑动窗口** | 固定步长的重叠窗口 | 信息完整、容错性好 | 产生大量冗余块 |

**设计原则：**
- **先语义、后大小**：优先保证每个块的内容逻辑完整，再控制大小
- **块大小与检索需求匹配**：如果用户问题通常是具体的事实查询（"退货政策是什么"），块可以小一些（256 Token）；如果是需要理解的综合问题（"分析这份报告的核心观点"），块需要大一些（1024+ Token）
- **元数据携带**：每个块携带来源文档、页码、章节标题等元数据，便于引用溯源

#### 3.3 分块实验指南

建议为每个 RAG 产品做一次分块策略 A/B 测试：
1. 准备 3-5 种分块方案（不同大小 + 不同策略）
2. 准备 100 个代表性测试问题（含难例和简单例）
3. 分别跑检索，比较 recall@k 和 precision@k
4. 人工评估 Top-3 块对回答问题的帮助程度

### 模块 4：检索策略

**核心问题：如何从知识库中找到最相关的内容？**

#### 4.1 检索方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **向量检索** | 用 Embedding 相似度（cosine/IP） | 语义匹配，理解意图 |
| **关键词检索** | BM25 / 倒排索引 | 精确匹配、专业术语 |
| **混合检索** | 向量 + 关键词加权融合 | 兼顾语义和精确匹配（推荐 ✅） |
| **重排序 (Rerank)** | 第一阶段粗筛，第二阶段精排 | 提高精度，推荐用于 Top-100 以上的检索 |
| **多轮检索** | 先检索一次，根据结果再检索 | 复杂查询需要多步获取信息 |

#### 4.2 混合检索权重设计

```
最终分数 = α × 向量相似度 + (1-α) × 关键词匹配度
```

- **α = 0.7-0.9**：语义优先（适合开放域问答）
- **α = 0.3-0.5**：关键词优先（适合术语密集型场景，如法律、技术文档）
- **动态 α**：根据查询类型动态调整（短查询偏向语义，长查询偏向关键词）

#### 4.3 Rerank 策略

| 策略 | 说明 | 额外延迟 | 效果提升 |
|------|------|----------|----------|
| 无 Rerank | 直接返回向量检索 Top-K | 0 | 基线 |
| 轻量 Rerank | Cohere Rerank / BGE-Reranker | +20-50ms | Recall@5 提升 5-15% |
| LLM Rerank | 用 LLM 判断相关性并排序 | +200-500ms | Recall@5 提升 10-25% |

**设计原则：**
- **检索 "多取少用"**：第一阶段检索更多的候选（如 Top-30），Rerank 后精选 Top-3 到 Top-5 送入 LLM
- **查询重写 (Query Rewrite)**：用户的问题可能是模糊的，先用小 LLM 改写为更精确的检索查询
- **混合检索是标配**：纯向量检索和纯关键词检索都有缺陷，混合策略是最安全的默认选择

### 模块 5：上下文窗口

**核心问题：哪些检索结果送入 LLM？如何组织上下文？**

#### 5.1 上下文组装策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **单层 K 块** | 直接取 Top-K 块拼入上下文 | 简单问答 |
| **分层检索** | 先检索粗粒度块，再在块内检索细粒度内容 | 长文档分析 |
| **摘要式** | 对检索结果做摘要再送入 LLM | 大量检索结果需压缩 |
| **对话历史 + 检索** | 结合历史对话和当前检索结果 | 多轮对话 |
| **多源融合** | 从不同知识库分别检索，按优先级排列 | 多数据源场景 |

#### 5.2 上下文窗口管理

| 考虑点 | 说明 |
|--------|------|
| **Token 预算** | 给检索结果分配多少 Token？通常占总上下文的 50-70% |
| **结果排序** | 按相关性从高到低排列，防止 LLM 被低质量内容干扰 |
| **引源标注** | 每个检索结果标注来源，便于 LLM 引用和用户验证 |
| **去重** | 检索结果可能包含相似/重复内容，需要去重后再送入 LLM |
| **权限标记** | 不同权限级别的文档做标记，LLM 根据用户权限过滤 |

#### 5.3 上下文组装模板示例

```markdown
请基于以下参考信息回答用户问题。

参考信息：
[1] (来源: 产品手册 v2.3, 第12页) 退货政策：……
[2] (来源: FAQ 文档) 退款流程：……
[3] (来源: 客服日志-2025) 常见问题：……

用户问题：
{user_query}

请用中文回答。如果参考信息不足以回答问题，请明确说明。回答时请标注引用来源 [1][2]。
```

### 模块 6：生成策略

**核心问题：LLM 如何基于检索结果生成回答？**

#### 6.1 生成参数调优

| 参数 | 说明 | RAG 场景推荐值 | 效果影响 |
|------|------|----------------|----------|
| **temperature** | 随机性控制 | 0.1-0.3（偏低） | 低温度让回答更贴近检索结果 |
| **top_p** | 采样池大小 | 0.1-0.3 | 配合低 temperature |
| **max_tokens** | 最大输出长度 | 视问题复杂度定，通常 512-2048 | 过长可能引入幻觉 |
| **presence_penalty** | 话题重复惩罚 | 0.0-0.3 | 防止过度重复引用 |

#### 6.2 生成策略设计

| 策略 | 说明 |
|------|------|
| **引用强制** | 要求 LLM 在回答中标注引用来源 [1][2] |
| **未知坦诚** | 检索结果不足时，LLM 必须说"无法从现有知识库中找到相关信息"，而非自行编造 |
| **分点回答** | 面对多部分问题，LLM 按点逐一回答并引用 |
| **立场中立** | LLM 基于检索结果客观回答，不加入个人观点 |
| **原文引用** | 对关键事实直接引用原文，而非转述，降低幻觉风险 |

#### 6.3 生成后处理

- **引用验证**：验证 LLM 引用的块编号与实际检索结果一致
- **事实校检**：使用另一个 LLM 或规则引擎检查回答与检索结果的一致性
- **格式规范**：统一回答格式（如 Markdown 渲染、列表格式）

### 模块 7：评估方式

**核心问题：如何衡量 RAG 产品的好坏？**

RAG 产品的评估分为检索质量和生成质量两个维度。

#### 7.1 检索质量评估

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **Recall@K** | 正确答案在前 K 个结果中的比例 | Recall@5 ≥ 0.85 |
| **Precision@K** | 前 K 个结果中相关结果的比例 | Precision@5 ≥ 0.70 |
| **MRR (Mean Reciprocal Rank)** | 第一个正确答案的排名的倒数平均 | MRR ≥ 0.80 |
| **NDCG@K** | 归一化折损累计增益，考虑排序位置 | NDCG@10 ≥ 0.75 |

#### 7.2 生成质量评估

| 维度 | 评估方式 | 说明 |
|------|----------|------|
| **忠实度 (Faithfulness)** | 回答是否严格基于检索结果 | LLM-as-Judge 评分（1-5） |
| **有用性 (Helpfulness)** | 回答是否解决了用户问题 | LLM-as-Judge + 人工抽样 |
| **完整性 (Completeness)** | 是否覆盖了问题所有方面 | 人工标注 + 自动检测 |
| **引用准确性** | 引用来源是否与内容一致 | 规则校验 |
| **无幻觉率** | 没有编造不在检索结果中的信息 | 自动检测 + 人工验证 |

#### 7.3 RAGAS 评估框架

推荐使用 RAGAS（Retrieval Augmented Generation Assessment）标准评估框架：

| 指标 | 测量内容 | 计算方式 |
|------|----------|----------|
| **Faithfulness** | 回答是否忠实于检索结果 | 将回答拆解为 claim，检查每个 claim 是否在检索结果中被支持 |
| **Answer Relevance** | 回答是否与问题相关 | 用 LLM 生成问题和回答的相关性评分 |
| **Context Precision** | 检索结果中的相关条目比例 | 检查检索结果中的有用信息占比 |
| **Context Recall** | 所需信息是否都被检索到 | 检查回答对应的信息是否在检索结果中 |

#### 7.4 评估 Pipeline 建议

```yaml
评估流程：
1. 准备测试集：100-500 个 QA 对，覆盖常见问题 + 难例 + 边界情况
2. 离线评估：自动计算 RAGAS 指标 + 检索指标
3. 人工抽样：每周人工评估 50 条，关注质量而非数量
4. 线上监控：用户赞/踩、反馈、转人工率
5. 回归测试：每次模型/策略变更后跑全量测试集
```

### 模块 8：延迟优化

**核心问题：如何让 RAG 产品更快？**

延迟是 RAG 产品的用户体验关键指标。用户期望通常在 1-3 秒内获得回答。

#### 8.1 延迟分解

```
典型 RAG 请求延迟分布：
查询改写    ← 50-150ms
向量检索    ← 20-100ms (取决于向量库和索引规模)
Rerank      ← 20-200ms (取决于模型)
上下文组装  ← <10ms
LLM 生成    ← 500-5000ms (取决于模型大小和输出长度)
后处理      ← <50ms
───────────────────────────────────────────────
总计        ← 600-5500ms
```

#### 8.2 优化策略

| 优化方向 | 策略 | 预期效果 |
|----------|------|----------|
| **索引优化** | HNSW 索引参数调优、IVF 量化和量| 检索延迟降低 50-80% |
| **缓存** | 常见问题缓存、Embedding 缓存、检索结果缓存 | 热门查询响应 < 200ms |
| **流式输出** | LLM 输出使用 Server-Sent Events（SSE）流式传输 | 首 Token 时间 < 500ms |
| **预检索** | 用户输入时即开始检索（猜测用户意图） | 减少感知延迟 300-500ms |
| **模型选型** | 使用推理速度更快的模型（如 DeepSeek-V2-Lite vs V3） | 生成延迟降低 2-5x |
| **并行化** | 多个知识源同时检索 | 多源检索时间从串行变为最大源时间 |
| **精简上下文** | 控制送入 LLM 的上下文数量（Top-3 vs Top-10） | 每减少一块节省约 200-500ms |
| **KV Cache** | LLM 推理的 KV Cache 优化 | 相同前缀的请求加速 30-50% |

#### 8.3 延迟 SLA 设计

| 场景 | 目标延迟 | 策略 |
|------|----------|------|
| 实时聊天 | < 2s | 流式输出 + 缓存 + 小模型 |
| 后台分析 | < 5s | 全量检索 + Rerank |
| 批量处理 | 无实时要求 | 使用最大精度策略 |
| 移动端 | < 1.5s | 端侧检索 + 云端生成 |

## 三、完整案例：企业内部知识库 RAG 问答系统

### 产品背景
为一家 5000 人科技公司搭建的内部 HR & IT 知识库问答系统，涵盖员工手册、IT 支持文档、福利政策和内部工具使用指南。

### 画布填写

| 模块 | 内容 |
|------|------|
| **场景定义** | 员工自助查询 HR 政策和 IT 支持；减少 HR/IT 团队 40% 的重复性问题 |
| **知识库设计** | 源文档：500+ 篇知识库文章（Markdown + PDF）；更新频率：每周更新 10-20 篇；向量库：Qdrant 自托管；Embedding：bge-large-zh-v1.5 |
| **分块策略** | 递归分块，按标题→段落→句子层级切分；块大小 512 Token，重叠 50 Token；携带元数据（文档标题、分类标签、更新时间） |
| **检索策略** | 混合检索（向量 + BM25），α=0.7；Top-30 初筛 → BGE-Reranker 重排序 → Top-5 送入 LLM；支持按部门标签过滤（不同组织可查看不同政策） |
| **上下文窗口** | Top-5 块（约 2500 Token）+ 系统 prompt（500 Token）；引用强制标注来源；对话历史保留最近 3 轮 |
| **生成策略** | Temperature 0.2；引用格式 [文档标题]；当信息不足时回答"建议联系 HR 专员"并提供联系方式 |
| **评估方式** | 离线：Recall@5 ≥ 0.90，Faithfulness ≥ 4.5/5；在线：用户点赞率 ≥ 85%，转人工率 < 15%；每周人工评估 100 条 |
| **延迟优化** | 流式输出，首 Token < 1s；常用问题缓存（Top-200 问题命中率 ~30%）；HNSW 索引 ef_search=128 |

### 实际效果
- 员工自助解决率从 30% 提升到 78%
- HR/IT 工单量减少 52%
- 用户满意度 4.3/5
- 平均响应时间 1.8s

## 四、RAG 产品常见陷阱

| 陷阱 | 现象 | 解决方案 |
|------|------|----------|
| **分块不合理** | 检索结果语义不完整，回答断章取义 | 改用语义分块策略，增加块重叠 |
| **检索噪音大** | 检索结果和查询不相关，LLM 被干扰 | 引入 Rerank、调整检索参数、增加元数据过滤 |
| **Lost in the Middle** | LLM 忽略中间位置的检索结果 | 把最相关的结果放在开头和结尾 |
| **过度依赖检索** | 检索结果本身就错了，但 LLM 盲目采纳 | 引入事实验证、设置置信度阈值 |
| **Embedding 漂移** | 更新数据后分布变化，检索效果下降 | 定期监控检索指标，重建索引 |
| **权限泄露** | 用户检索到不应看到的文档 | 检索时做权限过滤（元数据过滤、用户身份注入） |

## 五、RAG 产品设计原则总结

| 原则 | 说明 |
|------|------|
| **📚 知识质量 > 检索技巧** | 知识库干净、结构化、更新及时，比任何检索优化都重要 |
| **🔍 检索 "多取少用"** | 粗筛多取，精排少用，质量优先 |
| **✂️ 分块即架构决策** | 分块策略决定了知识组织的底层逻辑，值得投入时间实验 |
| **📏 评估驱动迭代** | 没有量化指标的 RAG 产品是盲人摸象 |
| **⚡ 延迟即体验** | 缓存和流式是 RAG 产品的标配优化 |
| **📖 引用不可省略** | 用户需要能验证答案的来源 |
| **🔄 监控飞轮** | 用用户反馈持续改进检索和生成质量 |
