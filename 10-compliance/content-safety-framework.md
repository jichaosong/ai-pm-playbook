# AI 产品内容安全框架

> AI 产品内容安全的独特性：不是网络安全，是内容生成安全。模型本身可能产生有害内容，且一次安全疏漏影响数百万用户。

---

## 1. AI 内容安全的独特性

### 1.1 vs 传统网络安全

| 维度 | 传统网络安全 | AI 内容生成安全 |
|------|-------------|----------------|
| 攻击面 | 系统漏洞、网络攻击 | Prompt 注入、对抗性输入 |
| 风险来源 | 外部攻击者 | 模型自身 + 用户输入 |
| 输出问题 | 数据泄露 | 有害内容、幻觉、偏见 |
| 防御方式 | 防火墙、WAF、权限控制 | 输入/输出过滤、对齐训练 |
| 问责对象 | 安全团队 | 产品经理 + 算法团队 |

### 1.2 AI 特有风险分类

| 风险类型 | 说明 | 风险等级 |
|---------|------|---------|
| 违法内容 | 暴力恐怖、儿童剥削、毒品制作 | 极高 |
| 仇恨言论 | 种族/性别/宗教攻击 | 高 |
| 色情内容 | 露骨色情、色情角色扮演 | 高 |
| 自残/自杀 | 鼓励或指导自残行为 | 极高 |
| 危险行为 | 武器制造、黑客工具、欺诈指南 | 高 |
| 医疗误诊 | 未经审核的医疗建议 | 极高 |
| 金融误导 | 投资建议、虚假理财信息 | 高 |
| 法律误导 | 法律意见替代、合同风险评估 | 高 |

### 1.3 为什么要求更高？

1. **规模化风险**——一个模型同时服务数百万用户
2. **无意图性**——模型不是故意犯错，但错误更隐蔽
3. **边界模糊**——"有害"在不同文化、地区定义不同
4. **对抗性攻击**——用户可通过精心设计的 Prompt 绕过安全限制
5. **合规压力**——全球 AI 监管日趋严格（EU AI Act、中国生成式AI管理办法等）

## 2. 产品安全架构设计

### 2.1 三层架构

```
用户输入 → 输入过滤层(Prompt安全检测+敏感词拦截)
              ↓ 通过
            模型推理
              ↓ 输出
            输出过滤层(关键词+语义+分类器)
              ↓ 通过
            审核队列(高风险→人工)
              ↓
            用户看到
```

### 2.2 输入过滤设计

| 层级 | 检测方法 | 拦截策略 | 延迟 |
|------|---------|---------|------|
| L0 - 模式匹配 | 正则表达式、关键词列表(10万+) | 直接拒绝 | < 1ms |
| L1 - 分类器 | 轻量 ML 模型(如 BERT) | 拦截+修改建议 | 10-50ms |
| L2 - 语义检测 | Moderation API | 拦截+上报+记录 | 100-500ms |
| L3 - 人工前置 | 高风险客户需人工审核 | 审核通过后放行 | 分钟级 |

**Prompt 注入检测**：
- 指令覆盖检测——试图覆盖系统提示词
- 越狱模板匹配——已知 jailbreak 模式库
- 分隔符检查——使用特殊标记绕过
- 角色扮演检测——试图让模型扮演其他角色
- 递归检测——多层嵌套复杂注入

**拦截用户反馈**：
```
您的输入触发了安全检测，无法处理。请修改后重试。
可能的原因：包含了不当/敏感内容，或试图让模型执行超出范围的操作。
如需申诉请点击：申诉入口
```

### 2.3 输出过滤设计

| 检测维度 | 方法 | 触发动作 | 误报处理 |
|---------|------|---------|---------|
| 关键词匹配 | 黑名单词库 | 整段替换为占位符 | 白名单机制 |
| 语义分类 | 内容分类模型 | 标记高风险→进入审核 | 置信度阈值调整 |
| 实体检测 | NER 模型 | 脱敏敏感实体(身份证/电话) | 上下文保留 |
| 情感分析 | 情感分类 | 极端负面/仇恨内容标记 | 场景差异化 |
| 一致性检查 | 对比输入/输出 | 偏离主题的异常输出标记 | N/A |

**输出修改策略**：

| 策略 | 说明 | 适用场景 | 用户体验影响 |
|------|------|---------|-------------|
| 彻底拦截 | 不展示任何输出 | 高风险违法内容 | 用户困惑 |
| 部分屏蔽 | 只屏蔽问题段落 | 局部有害内容 | 较好 |
| 替换内容 | 用安全内容替代 | 规避简单关键词 | 中等 |
| 降级回答 | 提供简化或泛化回答 | 边界场景 | 可接受 |
| 追加免责 | 回答后加免责声明 | 低风险误导场景 | 最小 |

### 2.4 分级安全策略

| 等级 | 业务场景 | 容忍度 | 过滤强度 | 审核要求 |
|------|---------|-------|---------|---------|
| S1 - 极低风险 | 娱乐、闲聊、创作辅助 | 高 | 低 | 无人工审核 |
| S2 - 低风险 | 教育、学习辅导 | 中 | 中 | 采样审核 |
| S3 - 中风险 | 客户服务、信息检索 | 中低 | 高 | 自动+抽审 |
| S4 - 高风险 | 金融咨询、法律建议 | 低 | 极高 | 全部人工审核 |
| S5 - 极高风险 | 医疗诊断、药物推荐 | 近零 | 最严格 | 多人审核+专家确认 |

**示例**：
- **娱乐(S1)**：仅基础关键词过滤，语义分类器阈值0.9，无人工
- **教育(S2)**：关键词+轻量分类器，阈值0.8，随机采样1%人工
- **金融(S4)**：全量多层过滤+语义+实体+一致性检查，高风险100%人工

## 3. 幻觉管控的产品策略

### 3.1 什么是 AI 幻觉

模型生成看似合理但实际错误或无根据的内容。这是 AI 产品最常见且最难彻底解决的安全问题。

### 3.2 四层管控策略

| 层级 | 措施 | 负责团队 |
|------|------|---------|
| 技术层 | RAG、知识图谱约束、事实一致性训练、拒绝回答训练 | 算法 |
| **产品层** | **置信度标识、来源引用、知识边界限制、免责声明** | **PM** |
| 交互层 | 不确定性表达、建议验证引导、用户反馈机制 | UX |
| 流程层 | 内容抽样审核、问题反馈闭环、持续改进机制 | 运营 |

### 3.3 置信度标识

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 显式分数(百分比) | 精准 | 用户理解困难 | 金融/医疗 |
| 等级(高/中/低) | 直观 | 粒度粗糙 | 通用 |
| 颜色编码(绿/黄/红) | 一目了然 | 色盲不友好 | 教育/客服 |
| 图标(✓/⚠/✗) | 视觉友好 | 需图例 | C端 |

**示例**：
```
用户：量子计算在金融领域有哪些应用？
回答：[高置信度 ✓] 量子计算在金融领域的应用包括：
1. 投资组合优化——量子算法可同时评估...
2. 风险分析——量子蒙特卡洛模拟...
📚 来源：[McKinsey 2024 Quantum Computing Report]

用户：昨天比特币为什么涨了？
回答：[低置信度 ⚠] 我无法获知今天的新闻。
短期波动受以下因素影响：宏观经济政策、市场情绪...
建议查看最新金融新闻获取准确信息。
```

### 3.4 来源引用

| 场景 | 引用要求 | 实现方式 |
|------|---------|---------|
| 娱乐/闲聊 | 不强制 | 可选显示 |
| 教育 | 推荐引用 | 自动添加来源 |
| 信息检索 | 强制引用 | RAG 显式标注 |
| 金融 | 强制引用+时效标注 | 数据源+日期 |
| 医疗 | 强制引用+免责声明 | 建议咨询医生 |

### 3.5 知识边界限制

**设定范围**：
```
知识截止日期：{YYYY-MM-DD}
专业领域：通用知识、{产品定义领域}
法律/医疗/金融：仅提供基本信息，不构成专业意见
实时信息：需要联网搜索功能
```

**超出边界回应**：
- 知识过时：`"我的知识截止于{日期}，建议开启联网搜索或访问以下网站"`
- 超出范围：`"建议咨询持有相应资质的专业人士"`
- 无法确认：`"为了不提供不准确信息，我暂时无法回答，请提供更多上下文"`

### 3.6 免责声明策略

| 场景 | 内容 | 展示位置 | 是否可跳过 |
|------|------|---------|-----------|
| 医疗 | "我不是医生，请咨询专业医疗人员" | 回答末尾 | 不可跳过 |
| 法律 | "不构成法律意见" | 回答开头 | 不可跳过 |
| 金融 | "不构成投资建议" | 回答末尾 | 不可跳过 |
| 实时信息 | "信息来自公开来源，请核实" | 回答末尾 | 普通 |
| 代理决策 | "最终决定权在您手中" | 操作确认前 | 弹窗确认 |

## 4. 内容审核流程

### 4.1 审核架构

```
用户输入/输出 → 自动审核引擎(ML分类器+规则引擎)
                 ├── 通过(≥90%) → 直接展示
                 ├── 存疑(5-8%) → 人工审核队列
                 └── 拦截(1-2%) → 拒绝展示+记录日志
```

### 4.2 自动审核引擎

| 层级 | 方法 | 延迟 | 覆盖率 |
|------|------|------|--------|
| L0 | 规则引擎(敏感词10万+/正则/黑白名单/频率限制) | 实时 | 全量 |
| L1 | 轻量分类器(文本分类~50ms/PII检测/情感分析) | 实时 | 全量 |
| L2 | 深度语义分析(Moderation API/多模态/上下文关联) | 准实时 | 采样 |
| L3 | 知识图谱校验(关系验证/事实一致性/逻辑矛盾) | 准实时 | 采样 |

### 4.3 人工审核设计

**审核工作台功能**：
- 审核队列(优先级排序+批量操作+筛选搜索)
- 审核界面(用户输入+模型输出+自动审核结果+用户历史+相似案例)
- 审核操作(通过/驳回/修改/升级给高级审核员)
- 质量监控(一致性检查+绩效统计+错误复盘)

**SLA 标准**：

| 级别 | 时限 | 审核员配置 | 超时处理 |
|------|------|-----------|---------|
| P0 - 紧急 | < 5分钟 | 专职24/7 | 自动降级策略 |
| P1 - 高 | < 30分钟 | 专职 | 升级组长 |
| P2 - 中 | < 4小时 | 兼职 | 自动通过 |
| P3 - 低 | < 24小时 | 轮值 | 自动通过 |

### 4.4 申诉机制

**流程**：拦截通知→用户提交申诉(原因选择+补充说明)→自动复审/AI二次判定/人工复审→结果通知

**核心指标**：

| 指标 | 定义 | 健康值 | 告警线 |
|------|------|-------|--------|
| 申诉率 | 申诉数/拦截数 | < 5% | > 10% |
| 申诉成功率 | 成功数/申诉数 | > 30% | < 15% |
| 处理时间 | 提交到结果 | < 2h | > 8h |
| 自动复审准确率 | 判定正确率 | > 95% | < 90% |

### 4.5 审核延迟与用户体验权衡

| 模式 | 延迟 | 安全等级 | 适用场景 |
|------|------|---------|---------|
| 实时同步 | < 1s | 中低 | 聊天、问答 |
| 准实时 | 1-5s | 中 | 教育、客服 |
| 异步审核 | 分钟级 | 高 | 内容生成、创作 |
| 前置审核 | 小时级 | 极高 | 高质量内容发布 |

**混合策略**：实时模式(输入L0-L1+输出L0-L1 <50ms)→用户看到内容→后台异步L2深度检查(500ms-2s)→发现问题时通知/撤回/标记。

## 附录A: 安全策略配置模板

### A.1 分层过滤阈值配置

```yaml
security_policy:
  input_filter:
    L0_keywords:
      enabled: true
      word_list_version: "2025-01"
      action: block
    L1_classifier:
      enabled: true
      model: "bert-toxicity-v3"
      threshold: 0.85
      action: block_with_hint
    L2_moderation:
      enabled: true
      provider: "internal-moderation-api"
      threshold: 0.75
      action: log_and_review

  output_filter:
    keyword_match:
      enabled: true
      partial_block: true
    semantic_classifier:
      enabled: true
      model: "content-safety-v2"
      threshold: 0.8
      action: flag_for_review
    pii_detection:
      enabled: true
      mask_mode: "partial"
```
