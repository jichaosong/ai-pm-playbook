# AI 产品从 0 到 1 完整流程 — 产品生命周期指南

> 面向 AI 产品经理，提供从问题定义到上线的端到端流程框架。基于 12 周标准时间线。

---

## 1. 概述与时间线总览

AI 产品与传统软件的核心区别：模型行为的不确定性。开发前无法 100% 确认模型在真实场景的表现。

| 维度 | 传统产品 | AI 产品 |
|------|---------|---------|
| 行为确定性 | 高（代码即行为） | 低（模型即行为） |
| 评估方式 | 功能测试通过/不通过 | 多维指标 + 人工评估 |
| 灰度风险 | 功能 bug | 模型行为异常 |
| PM 核心职责 | 需求 + 验收 | 需求 + 评估 + Prompt 迭代 |

### 12 周标准时间线

```
Week 1-2  │ 问题定义与用户验证
Week 3-4  │ 模型选型与 Prototype
Week 5-6  │ 评估体系建立
Week 7-8  │ MVP 开发
Week 9-10 │ 灰度与数据采集
Week 11-12│ 迭代与上线
```

各阶段有重叠，特别在 Week 5 就应开始准备评估数据。

## 2. Week 1-2：问题定义与用户验证

### 核心目标

确认三个问题：
1. **用户真的需要吗？** — 通过用户访谈验证
2. **AI 合适吗？** — 技术可行性判断
3. **值得做吗？** — 商业价值评估

### 用户访谈计划

| 访谈对象 | 目标 | 典型问题 |
|---------|------|---------|
| 目标用户（3-5 人） | 了解工作流痛点 | "当前怎么解决？" |
| 决策者（1-2 人） | 购买意愿和预算 | "愿意花多少钱？" |
| 行业专家（1-2 人） | 领域知识边界 | "专家和新人的差异？" |

### 假设验证框架

| 假设 | 验证方法 | 验证标准 |
|------|---------|---------|
| H1：用户每周花 5h+ 做这个 | 时间日志 + 访谈 | ≥70% 用户确认 |
| H2：用户愿为 AI 付费 | 价格敏感度测试 | ≥50% 愿付费 |
| H3：LLM 能完成该任务 | 模型快速测试 | 5 例通过 3 例 |
| H4：用户能接受当前准确性 | 用户对输出评分 | ≥4/5 分 |

### Go/No-Go 决策矩阵

| 维度 | 1 分 | 3 分 | 5 分 |
|------|------|------|------|
| 问题普遍性 | 小众 | 中等 | 大规模 |
| AI 优势 | 传统方案已够 | AI 略有优势 | AI 明显优于传统 |
| 付费意愿 | 不愿 | 小额 | 合理付费 |
| 数据可获得 | 无数据 | 需标注 | 现成数据 |
| 错误容忍度 | 零容忍 | 中等 | 高容忍 |

**决策规则**：总分 ≥ 20 → 绿灯；15-19 → 黄灯需验证；≤ 14 → 红灯暂缓

### 关键产出物

| 产出物 | 格式 | 说明 |
|--------|------|------|
| 用户访谈报告 | 文档 | 含关键洞察和用户原话 |
| 假设验证表 | 表格 | 所有假设及验证结果 |
| 机会评估文档 | 1 页纸 | 问题定义、目标用户、价值主张 |
| Go/No-Go 决策记录 | 文档 | 理由和风险分析 |

## 3. Week 3-4：模型选型与 Prototype

### 核心目标
1. 选择最合适的模型（不一定是最好）
2. 快速验证模型在真实任务上的表现
3. 确定技术方案边界

### 模型评估矩阵

| 模型 | 准确性 | P50 延迟 | 成本/1K tokens | 上下文 | 多模态 | 合规 |
|------|--------|---------|---------------|--------|--------|------|
| GPT-4o | ★★★★★ | 800ms | $0.005 | 128K | ✅ | 中 |
| Claude 3.5 Sonnet | ★★★★☆ | 700ms | $0.003 | 200K | ✅ | 低 |
| Gemini 1.5 Pro | ★★★★☆ | 600ms | $0.002 | 1M | ✅ | 中 |
| DeepSeek V3 | ★★★☆☆ | 1.2s | $0.001 | 128K | ❌ | 需评 |
| Qwen 2.5 | ★★★☆☆ | 500ms | $0.0005 | 128K | ✅ | 低 |

### Prototype 三种方法

| 方法 | 耗时 | 产出 |
|------|------|------|
| Jupyter Notebook | < 1 天 | 代码输出 + 人工评分 |
| Streamlit / Gradio | 1-2 天 | 可交互 Demo |
| 迷你后端 + UI | 3-5 天 | 接近产品体验的原型 |

### PM 在原型阶段的工作

| 工作 | 预计时间 |
|------|---------|
| 准备 20-50 个真实用户测试用例 | 1 天 |
| 编写初版 prompt | 1-2 天 |
| 人工评估模型输出 | 2-3 天 |
| 成本测算 | 0.5 天 |

### 关键产出物

- 模型选型报告（评估矩阵 + 推荐理由）
- Prototype 演示 Demo
- 技术可行性评估文档
- 成本估算表

## 4. Week 5-6：评估体系建立

### 离线评估集构建原则

- **代表性**：覆盖真实用户输入分布
- **边缘性**：边界情况（长文本、特殊格式、多语言）
- **异常性**：空输入、恶意输入、无关输入
- **稳定性**：评估集不频繁变动，便于版本对比

### 评估集规模建议

| 阶段 | 规模 | 说明 |
|------|------|------|
| Prototype | 20-50 条 | 快速筛选模型 |
| MVP | 100-200 条 | 覆盖主要场景 |
| 正式上线 | 500-1000 条 | 全面覆盖 |
| 持续迭代 | 2000+ 条 | 含回归测试 |

### 产品级 vs 技术级指标

| 类别 | 指标 | 目标值 | 负责人 |
|------|------|--------|--------|
| 产品级 | 任务完成率 | ≥85% | PM |
| 产品级 | 首次回答满意率 | ≥70% | PM |
| 产品级 | 平均交互轮次 | ≤3 轮 | PM |
| 技术级 | 准确率 | ≥90% | MLE |
| 技术级 | 幻觉率 | ≤5% | MLE |
| 技术级 | P50/P99 延迟 | ≤1s/≤3s | MLE |

### Benchmark 设计

每个测试用例包含：
```json
{
  "id": "BENCH-001",
  "input": "用户输入",
  "expected": "期望输出",
  "difficulty": "中等",
  "eval_criteria": ["准确性", "完整性"]
}
```

### PM 在评估期间的工作

| 工作 | 时间 |
|------|------|
| 编写评估用例 | 3 天 |
| 定义评分标准 | 1 天 |
| 组织标注 | 3 天 |
| 评审评估结果 | 2 天 |
| 更新 Prompt | 持续 |

## 5. Week 7-8：MVP 开发

### PM 在开发期间做什么

| 工作 | 时间占比 |
|------|---------|
| Prompt 迭代（基于评估结果） | 30% |
| 数据准备（标注数据、评估数据） | 20% |
| 用户体验设计（Error State、Loading） | 15% |
| 评估验收 | 15% |
| 编写用户文档、灰度计划 | 20% |

### Prompt 迭代流程

```
评估结果 → 识别主要问题 → 修改 Prompt v1.1 → 评估集验证 → 评分对比
    ↓                                                           ↓
    未达标 → 继续修改 ←←←←←←←←←←←←←←←←←←←←←←←←←←←←
    ↓
    达标 → 锁定 Prompt 版本
```

### 灰度前质量门禁

- [ ] 离线评估通过（所有产品级指标达标）
- [ ] 安全性审查通过
- [ ] 性能达标（P50 < 1s, P99 < 3s）
- [ ] 成本在预算范围内
- [ ] 内部使用 3 天无严重问题
- [ ] 用户访谈验证通过
- [ ] 数据埋点就绪
- [ ] 回滚方案就绪

## 6. Week 9-10：灰度与数据采集

### 四阶段灰度发布

```
阶段 1：内部灰度（1-2 天）→ 流量 1%
阶段 2：小范围灰度（2-3 天）→ 流量 5%
阶段 3：中范围灰度（3-5 天）→ 流量 20% → 50%
阶段 4：全量发布 → 流量 100%
```

### 用户分桶策略

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 按用户 ID hash | 稳定 | 冷启动不均匀 | 长期 A/B |
| 随机采样 | 简单 | 体验不一致 | 短期测试 |
| 按付费层级 | 控制风险 | 组间不可比 | 谨慎上线 |

### 必埋点

| 事件 | 触发条件 | 附带参数 |
|------|---------|---------|
| query_submit | 用户提交 | query_length, user_id |
| response_received | 模型返回 | latency, token_count, model_version |
| response_action | 用户操作 | copy/edit/regenerate/dislike |
| session_start/end | 会话起止 | duration, interaction_count |

### 灰度放量决策

- [ ] 核心指标无显著下降（p < 0.05）
- [ ] 错误率 < 0.5%
- [ ] 负面反馈率 < 2%
- [ ] 无 P0/P1 Bug
- [ ] 客服无批量投诉
- [ ] 成本在预算内

全部通过 → 放量。任一不通过 → 暂停。

### 用户反馈收集

| 方式 | 收集量 | 质量 |
|------|--------|------|
| 应用内 Feedback Widget | 高 | 中 |
| NPS 问卷 | 中 | 中 |
| 深度访谈（每周 2-3 人） | 低 | 高 |
| 客服数据 | 持续 | 中 |
| 行为分析 | 高 | 中 |

## 7. Week 11-12：迭代与上线

### 基于数据的迭代

灰度数据 → 问题分类 → 优先级排序 → 修改 → 验证 → 上线

**常见问题类型：**

| 问题 | 数据表现 | 修复方案 |
|------|---------|---------|
| Prompt 理解偏差 | 回答偏离意图 | 优化 Prompt |
| 知识覆盖不足 | 频繁答"不知道" | 增加 RAG |
| 格式问题 | 用户频繁编辑 | 严格格式要求 |
| 过度自信 | 回答错但看起来很对 | 加入不确定性表达 |
| 延迟敏感 | 会话放弃率高 | 优化推理或降级模型 |

### A/B 测试决策树

```
实验组显著优于对照组 → 全量切换到 B
实验组与对照组无差异 → 成本更低则切换，更高则保持
实验组显著差于对照组 → 分析原因，修复再测
```

### 上线决策矩阵

| 条件 | 决策 |
|------|------|
| 所有指标达标 | ✅ 正式上线 |
| 核心达标，次要下降 | ⚠️ 有条件上线，制定修复时间表 |
| 核心未达标 | ❌ 暂缓上线 |
| 安全/合规问题 | 🚫 立即回滚 |

### 上线 Checklist

- [ ] 最终评估报告完成
- [ ] A/B 测试结果已分析
- [ ] 上线决策获各方确认
- [ ] 回滚方案就绪
- [ ] 监控告警已配置
- [ ] 客服已通知
- [ ] 用户通知已准备
- [ ] 避开高峰期

### 上线后监控节奏

| 时间点 | 动作 |
|-------|------|
| 1 小时 | 核心指标检查，异常立即回滚 |
| 24 小时 | 全面指标检查 |
| 1 周 | 用户行为分析 |
| 1 个月 | ROI 评估，路线图更新 |

## 8. 常见时间线陷阱

### 陷阱 1：模型评估时间不足
**表现**：1 天选模型，后来发现不满足需求，重新选型 → 2-3 周 delay
**预防**：评估 3-5 个模型，每个用 20 个真实用例测试

### 陷阱 2：数据准备超时
**表现**：Week 5 开始准备数据，Week 7 发现质量不够 → 1-2 周 delay
**预防**：Week 1 就启动数据采集，标注指南提前写好

### 陷阱 3：灰度发现问题太多
**表现**：每天 5-10 个严重问题，灰度无限延长 → 3-4 周 delay
**预防**：灰度前充分内部试用，建立问题优先级标准，设灰度最大时长

### 陷阱 4：PM 在开发期无事可做
**表现**：PM 等开发完成才介入 → Prompt 不合适，评估体系不完善
**预防**：PM 持续迭代 Prompt，提前准备灰度计划

### 陷阱 5：指标过多无法决策
**表现**：20+ 个指标都有波动 → 决策瘫痪
**预防**：确定 3 个北极星指标，其他仅作参考

### 陷阱 6：上线后缺乏持续投入
**表现**：产品上线后团队转向新项目 → 体验停滞
**预防**：上线后留 2 周持续优化，模型迭代纳入常规节奏
