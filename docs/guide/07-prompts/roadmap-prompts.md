## 模板 1：从零开始生成产品路线图

**使用场景**：新产品或新方向的路线图规划，需要确定产品从 MVP 到成熟版本的迭代路径。

**提示词**：

```markdown
你是一位资深 AI 产品经理，请帮我在[产品方向]上制定一份从零到一的产品路线图。

## 产品背景
- 产品定位：[一句话描述你的 AI 产品]
- 目标用户：[目标用户群]
- 核心价值：[用户的核心收益]
- 关键约束：[如：6 个月内上线、3 人团队、预算有限]

## 请产出以下内容

### 1. 产品愿景（12-24 个月后的目标）
- 一句话愿景
- 成功后的产品形态描述
- 目标市场规模和用户量的预估

### 2. 分阶段路线图

**Phase 0：验证期（第 1-2 月）**
- 目标：验证核心假设
- 交付物：原型 / Demo / 用户调研报告
- 关键决策点：是否继续

**Phase 1：MVP 期（第 3-4 月）**
- 目标：最小可行产品上线
- 核心功能清单（限制在 3-5 个核心功能）
- 评估指标和成功标准
- 风险和缓解措施

**Phase 2：增强期（第 5-8 月）**
- 目标：基于用户反馈完善产品
- 新增功能清单
- 优化方向（性能、体验、准确性）
- 开始积累数据和了解用户行为

**Phase 3：规模化期（第 9-12 月）**
- 目标：扩大用户规模，验证商业模式
- 关键功能
- 商业化策略（如果适用）
- 基础设施和团队扩展需求

### 3. 每个阶段需要回答的关键问题
- 这个阶段最大的不确定性是什么？
- 如何验证这个阶段的核心假设？
- 什么条件下可以进入下一阶段？

### 4. 风险雷达
- 技术风险
- 市场风险
- 团队风险
- 竞争风险

注意：路线图是"活的文档"。每个阶段的计划都是假设，需要根据实际情况调整。标注出每个阶段的关键验证节点和决策点。
```

## 模板 2：AI 产品里程碑规划

**使用场景**：已有产品方向，需要细化为具体的时间线和可交付里程碑。

**提示词**：

```markdown
你是一位 AI 产品经理，请帮我为[产品名]制定详细的里程碑规划。

## 产品信息
- 当前状态：[如：已上线 MVP / 正在开发中 / 概念阶段]
- 目标：[如：Q3 上线完整版 / 年底达到 10 万用户]
- 团队资源：[如：2 名算法、3 名工程、1 名设计、1 名 PM]

## 请按季度规划里程碑

### 202X Q1（当前季度）
**主题**：[如：夯实基础]
- 核心目标（1-2 个 OKR）
- 关键里程碑（具体日期和交付物）
  - M1 (MM/DD)：[可交付物]
  - M2 (MM/DD)：[可交付物]
  - M3 (MM/DD)：[可交付物]
- 风险和依赖
- 资源分配计划

### 202X Q2
**主题**：[如：用户增长]
- 同上结构

### 202X Q3
**主题**：[如：商业验证]
- 同上结构

### 202X Q4
**主题**：[如：规模化]
- 同上结构

### 里程碑之间的依赖关系
- 哪些里程碑有前置依赖
- 关键路径（CPM）
- 如果有延期，优先保证哪些里程碑

### 每个里程碑的验收标准
- 明确什么算"完成"
- 谁来做验收
- 验收不通过怎么办

注意：里程碑要有明确的"完成"标准，不能是模糊的"做得更好"。每个里程碑都要问：这个交付物是否独立有价值？
```

## 模板 3：基于 OKR 的季度路线图

**使用场景**：结合公司和团队 OKR，将目标拆解为具体的产品路线图。

**提示词**：

```markdown
你是一位 AI PM，需要在 OKR 框架下制定季度产品路线图。

## 团队 OKR
- Objective 1：[描述]
  - Key Result 1：[量化指标]
  - Key Result 2：[量化指标]
- Objective 2：[描述]
  - Key Result 1：[量化指标]
- Objective 3：[描述]
  - Key Result 1：[量化指标]

## 请将 OKR 拆解为产品路线图

### 1. OKR 到产品的映射
| OKR | 产品策略 | 核心假设 | 验证方法 |
|-----|---------|---------|---------|
| [KR] | [产品策略] | [假设] | [验证方法] |

### 2. 季度产品路线图

**Month 1：探索与验证**
- 要做的事（3-5 个关键任务）
- 对应的 KR
- 预期产出
- 验证节点

**Month 2：构建与测试**
- 同上

**Month 3：发布与评估**
- 同上

### 3. 每个 KR 的追踪方式
- 数据指标定义
- 数据来源
- 复盘频率
- 谁负责追踪

### 4. 风险和对冲
- 如果某个 KR 进度落后怎么办
- 哪些方向可以调整
- 什么情况下需要和上级沟通调整 OKR

注意：PM 的核心工作是确保产品行动与 OKR 对齐。每个路线图上的任务都要能回答"这对应哪个 KR？"
```

## 模板 4：技术驱动 vs 需求驱动的路线图评估

**使用场景**：当 AI 技术快速发展时，需要平衡"技术能做什么"和"用户需要什么"来制定路线图。

**提示词**：

```markdown
你是一位 AI 产品策略师。在 AI 行业，路线图常常需要在"技术驱动"和"需求驱动"之间做平衡。请帮我评估[产品]的路线图策略。

## 背景
- 产品：[描述你的产品]
- 当前阶段：[探索期 / 增长期 / 成熟期]
- 技术环境：[如：模型能力正在快速提升 / 技术相对成熟]

## 评估框架

### 1. 需求驱动（Pull）的力量
请分析以下需求因素：
- 用户明确表达的痛点有哪些
- 市场数据（搜索趋势、竞品动态、行业报告）显示的机会
- 商业模式上的"必须做"
- 客户付费意愿最强的功能

### 2. 技术驱动（Push）的机会
请分析以下技术因素：
- 最近 3 个月出现了哪些值得关注的技术突破
- 哪些技术能力正在变得"足够好以商业化"
- 技术成本下降趋势（如 API 降价、开源模型成熟）
- 团队特有的技术优势

### 3. 平衡方案

| 时间 | 偏重 | 理由 | 具体行动 |
|------|------|------|---------|
| 短期 | [需求/技术] | [理由] | [具体] |
| 中期 | [需求/技术] | [理由] | [具体] |
| 长期 | [需求/技术] | [理由] | [具体] |

### 4. 需防范的风险
- 过度技术驱动：做出了没人用的功能
- 过度需求驱动：技术方案可能很快过时
- 错过技术窗口：等需求验证清楚了，技术优势已不在

### 5. 建议
请给出 3-5 条具体的路线图策略建议，平衡技术和需求的关系。

注意：不同阶段的 AI 产品应该有不同的平衡策略。MVP 阶段可以偏需求驱动，而平台型产品可能需要偏技术驱动保持竞争力。
```

## 模板 5：功能需求优先级排序（RICE 框架）

**使用场景**：需求池中有大量功能需求，需要用系统化的框架做优先级排序。

**提示词**：

```markdown
你是一位 AI 产品经理，请使用 RICE 框架对以下功能需求进行优先级排序。

## RICE 框架说明
RICE 分数 = (Reach × Impact × Confidence) / Effort

- **Reach**（覆盖范围）：在给定时间内（如一个季度），该功能会影响到多少用户
- **Impact**（影响力）：该功能对单个用户的影响程度
  - 3 = 极大影响（改变核心体验）
  - 2 = 高影响（显著改善）
  - 1 = 中等影响（一般改善）
  - 0.5 = 低影响（小改善）
  - 0.25 = 极小影响（微调）
- **Confidence**（信心）：你对上述评估的信心
  - 100% = 有充分数据支持
  - 80% = 有较强依据
  - 50% = 有经验判断
  - 20% = 纯猜测
- **Effort**（投入）：该功能需要多少人·月

## 功能需求列表
[列出功能需求，每个需求包含名称和简短描述]

## 请输出

### 1. RICE 评分表
| 功能 | Reach | Impact | Confidence | Effort | RICE 分数 |
|------|-------|--------|------------|--------|-----------|
| [功能名] | [数字] | [分数] | [%] | [人月] | [计算值] |
| ... | ... | ... | ... | ... | ... |

### 2. 优先级排序结果
- P0（必须做）：RICE 分数最高的 2-3 个
- P1（应该做）：后续 2-3 个
- P2（可以做）：剩余中值得做的
- P3（暂缓）：分数低的

### 3. AI 功能特别考量
对 AI 相关的功能，请在以上基础上补充：
- 技术可行性（模型能力是否足够）
- 数据可得性（需要的数据是否有）
- 评估可行性（能否有效评估效果）
- 安全/伦理风险

### 4. 争议说明
- 如果排序结果与直觉不符，请解释原因
- 标注 Confidence 较低的功能，提醒需要先验证

注意：RICE 分数是决策参考，不是最终决策。需要结合战略方向、技术依赖等因素做最终判断。建议每月/每季度重新评估一次。
```

## 模板 6：AI 能力成熟度路线图

**使用场景**：规划产品在 AI 能力上的渐进式提升路径，从简单规则到复杂 AI。

**提示词**：

```markdown
你是一位 AI 产品策略师。请帮我为[产品]制定一份 AI 能力成熟度路线图。

## 产品信息
- 产品：[产品名和简介]
- 当前 AI 能力：[如：还未接入 AI / 使用简单规则 / 已有简单 AI 功能]
- 愿景：[如：成为智能化的全方位助手]

## AI 能力成熟度模型

请将 AI 能力从 Level 0 到 Level 4 规划：

### Level 0：无 AI（当前状态）
- 产品当前的能力描述
- 用户如何完成任务
- 核心痛点

### Level 1：AI 辅助（3 个月内）
- AI 能力：简单规则 + 基础 LLM 调用
- 产品特征：
  - 用户可以手动触发 AI 功能
  - AI 提供建议，用户决定是否采纳
  - 功能范围有限、确定性高
- 技术栈建议
- 评估指标

### Level 2：AI 增强（3-6 个月）
- AI 能力：RAG / Fine-tuning / 简单 Agent
- 产品特征：
  - AI 功能嵌入核心流程
  - AI 能处理 60-70% 的常规情况
  - 用户主要对 AI 输出做审核和微调
- 技术栈建议
- 评估指标

### Level 3：AI 驱动（6-12 个月）
- AI 能力：多 Agent 协作 / 个性化 / 主动推荐
- 产品特征：
  - AI 能主动发起交互（如：主动推荐、预警）
  - AI 处理 85%+ 的常规任务
  - 用户角色转变为监督和例外处理
- 技术栈建议
- 评估指标

### Level 4：AI 自主（12-24 个月）
- AI 能力：自主学习 / 自我优化 / 全自动
- 产品特征：
  - AI 能自主学习和适应每个用户
  - 端到端的自动工作流
  - 用户只需设定目标和规则
- 技术栈建议
- 评估指标

### 每个阶段的升级条件
- 什么时候可以从 Level 1 进入 Level 2？
- 需要哪些数据积累？
- 需要哪些用户行为验证？

注意：并非所有产品都需要做到 Level 4。根据产品定位和用户需求，选择合适的 AI 成熟度终点。
```

## 模板 7：竞品对齐路线图分析

**使用场景**：发现竞品新增了关键 AI 功能，需要评估是否要对齐和如何差异化。

**提示词**：

```markdown
你是一位 AI 产品策略师。我们的竞品最近发布了新功能/新方向，请帮我分析这对我们的路线图意味着什么。

## 竞品动态
- 竞品名：[名称]
- 新发布内容：[描述新功能/新方向]
- 发布时间：[日期]
- 市场反应：[用户反馈、媒体报道、社区讨论]

## 我们的产品
- 产品：[名称]
- 当前路线图：[简要描述当前规划]

## 分析框架

### 1. 竞品功能分析
- 这个新功能解决了什么问题
- 技术实现难度评估
- 用户体验评价
- 它的弱点和不足在哪里

### 2. 对我们路线图的影响评估

| 影响维度 | 评估 | 说明 |
|---------|------|------|
| 竞争威胁 | [高/中/低] | [为什么] |
| 用户期望变化 | [高/中/低] | [为什么] |
| 技术差距 | [大/中/小] | [为什么] |
| 差异化空间 | [大/中/小] | [为什么] |

### 3. 应对策略选项

**选项 A：快速对齐**
- 在 X 个月内上线类似功能
- 需要的资源和代价
- 风险和收益

**选项 B：差异化竞争**
- 不在这个方向对齐，优先做竞品没有的
- 差异化方向建议
- 如何在用户心智中建立差异化

**选项 C：暂不行动**
- 为什么选择等待
- 我们通过什么其他方式满足用户需求
- 持续监控的指标

### 4. 路线图调整建议
- 如果调整路线图，需要改什么
- 如果不变，需要注意什么
- 下周/下月需要做什么

注意：不要为了对齐而对齐。先回答"我们的用户是否也需要这个？"再决定是否跟进。
```

## 模板 8：路线图展示与汇报建议

**使用场景**：需要向上级、团队或投资人展示产品路线图，清晰传达规划和理由。

**提示词**：

```markdown
你是一位 AI PM，需要准备一份产品路线图的汇报材料。请帮我设计展示内容和结构。

## 受众
- [如：副总裁/CTO/CEO/投资人/全体团队]
- 他们的关注点：[如：节奏是否合理 / 投入产出比 / 技术风险 / 团队负载]

## 路线图内容
- 当前产品状态：[描述]
- 下个季度计划：[描述]
- 全年规划：[描述]

## 请设计展示框架

### 1. 展示结构建议（10-15 分钟）
- 开场：为什么我们需要这个路线图（1 分钟）
- 这里（我们目前的位置）：现状和痛点（2 分钟）
- 那里（我们要去哪里）：愿景和目标（2 分钟）
- 怎么去（路线图）：分阶段规划和里程碑（5 分钟）
- 凭什么（信心来源）：数据、用户反馈、技术验证（2 分钟）
- 风险和后备计划（1 分钟）
- 需要什么支持（1 分钟）
- Q&A（2 分钟）

### 2. 每个部分的关键信息
列出每个部分必须传达的核心信息和精心准备的金句

### 3. 数据可视化建议
- 用什么图表达时间线（Gantt 图 / 时间轴 / 看板）
- 用什么方式呈现优先级（矩阵 / 气泡图 / 排序列表）
- 推荐的工具（Notion / Google Slides / Figma / Miro）

### 4. 常见问题预先准备
根据受众特点，提前列出可能被问的问题和准备回答：
- "为什么是这个时间点？"
- "为什么不先做 XX 功能？"
- "如果我们增加/减少资源会怎样？"
- "竞争对手呢？"
- "衡量成功的标准是什么？"

### 5. 展示技巧
- 如何管理会上讨论的方向偏移
- 如何针对不同受众调整语言
- 如何应对质疑而不陷入防御

注意：好的路线图汇报不只是展示计划，更是建立信任和争取支持的机会。展示要体现"思考深度"和"风险管理意识"。
```

## 路线图提示词使用建议

| 建议 | 说明 |
|------|------|
| **路线图是假设，不是承诺** | 明确标注每个阶段的假设条件和验证节点 |
| **AI 路线图要更灵活** | AI 技术演进快，建议按月而非按季度做检查 |
| **预留技术探索时间** | 在路线图中留出 20% 的时间做技术探索和原型 |
| **监控外部变化** | AI 行业变化快，定期（每月）审视竞品和技术趋势 |
| **向上沟通用愿景，向下沟通用计划** | 对高管讲"为什么"，对团队讲"怎么做" |
