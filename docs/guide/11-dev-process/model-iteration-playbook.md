# 模型升级的灰度策略与影响评估

> 模型升级是 AI 产品中最高风险的操作之一。一个"更强的模型"可能带来感知更差的用户体验。本文档提供系统化的灰度策略、影响评估框架和回滚机制。

---

## 1. 模型升级的风险认知

### 为什么模型升级是高风险操作

| 风险类型 | 表现 | 后果 |
|---------|------|------|
| 行为漂移 | 同样 Prompt 在不同模型输出不同 | 用户感知"产品变了" |
| 风格变化 | 语气、长度不可预测变化 | 品牌一致性受损 |
| 能力退化 | 新模型在某些维度比旧模型差 | 体验下降 |
| 延迟变化 | 推理时间变化（通常变慢） | 用户流失 |
| 成本变化 | Token 消耗或价格变化 | 成本超支 |
| 安全风险 | 安全防护能力不确定 | 合规事故 |

### 真实案例

```
某客服 AI 从 GPT-3.5 升级到 GPT-4

离线评估：准确率 +9%，完整率 +10% → 团队结论：明显更好
上线后：投诉率 +300%，重新生成率 +50%，放弃率 +30%

根因：GPT-4 回答平均多 40% tokens，用户觉得"管太宽"
用户习惯了简洁风格，新风格不适应

教训：离线评估的"准确率"不等于用户体验的提升。
```

### 风险分级

| 等级 | 升级类型 | 示例 | 建议灰度周期 |
|------|---------|------|------------|
| 低 | 同模型小版本 | GPT-4o 版本更新 | 1-3 天 |
| 中 | 同级别模型切换 | GPT-4o → Claude 3.5 | 5-7 天 |
| 高 | 模型架构切换 | GPT-3.5 → GPT-4 | 7-14 天 |
| 极高 | 自部署模型切换 | LLaMA 2 → LLaMA 3 | 14-21 天 |

## 2. 灰度策略

### 灰度总览

```
内部测试（1-5%）→ 小范围（5%）→ 中范围（20%→50%）→ 全量（100%）
```

### 阶段 1：内部测试

| 测试组 | 人数 | 内容 | 时长 |
|--------|------|------|------|
| MLE 团队 | 2-5 人 | 技术验证（延迟、格式、Token） | 1-2 天 |
| PM + 产品 | 3-5 人 | 产品体验评估 | 1-2 天 |
| 友好用户 | 5-10 人 | 真实场景测试 | 2-3 天 |

**检查清单：**
- [ ] 模型能否正常响应所有类型输入？
- [ ] 输出格式符合预期？
- [ ] 延迟 P50/P99 可接受？
- [ ] 回答风格符合品牌调性？
- [ ] 无新的安全漏洞？
- [ ] 对抗性测试通过？

### 阶段 2：小范围灰度（5% 流量，2-3 天）

**推荐分桶：按用户 ID hash**
- 桶 1（控制组 5%）：旧模型
- 桶 2（实验组 5%）：新模型
- 剩余 90%：旧模型

**监测指标：**

| 指标 | 与基线对比标准 |
|------|--------------|
| 负面反馈率 | ≤ 基线 + 1% |
| 重新生成率 | ≤ 基线 + 2% |
| 对话放弃率 | ≤ 基线 + 2% |
| P50 延迟 | ≤ 基线 + 500ms |
| P99 延迟 | ≤ 基线 + 2s |
| 错误率 | 无显著增加 |

**每日报告模板：**

```markdown
## 灰度日报 - Day X
范围：5% 实验组 vs 5% 控制组

| 指标 | 实验组 | 控制组 | 差异 | 状态 |
|------|--------|--------|------|------|
| 负面反馈 | 1.2% | 1.0% | +0.2% | ✅ |
| 重新生成 | 8.5% | 7.2% | +1.3% | ✅ |
| P50 延迟 | 1.2s | 0.8s | +0.4s | ✅ |

人工抽样（50 条）：4.2/5（旧 4.3/5）
主要问题：回答偏长（15 条）
结论：✅ 可推进到中范围
```

**放量条件：**
- [ ] 灰度 ≥ 2 天
- [ ] 核心指标无显著退化（p > 0.05）
- [ ] 无 P0/P1 Bug
- [ ] 人工抽样评分 ≥ 4/5
- [ ] 成本在预算内
- [ ] 延迟在可接受范围

### 阶段 3：中范围灰度（20%→50%，3-5 天）

**放量节奏：**
- Day 1：20% 用户 — 验证问题是否在更大规模复现
- Day 3：如正常 → 放量至 50%
- Day 5：如正常 → 全量决策

**核心关注问题：**
1. 用户是否注意到产品变了？（感知变化率）
2. 如果注意到了，觉得更好还是更糟？（满意度变化）
3. 用户行为有没有变化？（使用频率、留存）

**告警阈值：**
| 指标 | 阈值 |
|------|------|
| 显式负面反馈率 | > 基线 + 2% |
| 隐式负向信号率 | > 基线 + 3% |
| DAU 变化 | > 下降 5% |
| 人均使用时长 | > 下降 10% |
| 任务完成率 | > 下降 5% |

### 阶段 4：全量发布

**全量发布决策会议**（PM + MLE Lead + QA + 安全 + 客服）：
1. PM 汇总灰度数据（10min）
2. MLE 汇报技术指标（10min）
3. 安全/客服准备情况（10min）
4. 决策讨论（10min）

**发布前 Checklist：**
- [ ] 新模型服务完全部署，旧模型保持在线
- [ ] 监控告警已配置，回滚方案就绪
- [ ] 用户沟通文案已准备
- [ ] 内部 FAQ 已更新给客服
- [ ] 值班安排确定（上线后 48h）
- [ ] 紧急联系人已确认
- [ ] 发布时间选定（低流量时段）

## 3. 影响评估框架

### 三个评估层面

```
模型层（MLE）：准确率、幻觉率、格式合规、延迟
产品层（PM）：任务完成率、满意度、用户行为
业务层（PM+业务）：留存、转化、LTV、品牌口碑
```

### 正向影响评估

| 正向影响 | 衡量指标 | 数据来源 |
|---------|---------|---------|
| 回答质量提升 | 准确率、满意度 | 评估集、调研 |
| 任务完成率提升 | 成功完成任务比率 | 产品埋点 |
| 用户信任度提升 | 重新生成率下降 | 产品埋点 |
| 新能力覆盖 | 新增场景数量 | 功能测试 |

### 负向影响评估

| 负向影响 | 风险 | 评估方法 | 阈值 |
|---------|------|---------|------|
| 回答风格变化 | 中 | 用户感知调查 | >10% 用户感觉变化 |
| 性能延迟增加 | 高 | 延迟监控 | P50 > 2s 或 P99 > 5s |
| 成本增加 | 中 | 成本监控 | 超预算 20% |
| 功能退化 | 高 | 回归测试 | 任何之前能回答的不能了 |
| 安全违规 | 极高 | 安全测试 | 任何事件 |
| 满意度下降 | 高 | 反馈监控 | NPS 下降 > 5 分 |

### 影响评估报告模板

```markdown
## 升级影响评估：模型 A → 模型 B

### 正向影响
| 维度 | 旧 | 新 | 变化 | 显著 |
|------|----|----|------|------|
| 准确率 | 87.5% | 93.2% | +5.7% | p<0.01 |
| 任务完成率 | 72.3% | 78.9% | +6.6% | p<0.05 |

### 负向影响
| 维度 | 旧 | 新 | 变化 | 是否接受 |
|------|----|----|------|---------|
| 平均延迟 | 0.8s | 1.4s | +75% | ⚠️ 可接受 |
| Token 消耗 | 425 | 580 | +36% | ✅ 预算内 |
| 回答长度 | 85 字 | 132 字 | +55% | ⚠️ 关注 |

### 用户感知
感觉"变好了"：35% | 感觉"变差了"：12% | 感觉"没变化"：53%

### 结论
✅ 建议上线。收益大于风险，需跟进优化回答长度和延迟。
```

## 4. 用户感知评估

### 为什么需要专门评估用户感知

```
离线评估告诉你：模型是否"正确"
用户感知告诉你：用户是否"感觉"好

两者可能完全不同：
- 正确但太长 → "太啰嗦"
- 正确但太正式 → "冷冰冰"
- 正确但变了 → "不好用了"
```

### 评估方法

| 方法 | 成本 | 样本量 | 数据质量 |
|------|------|--------|---------|
| 应用内弹窗问卷 | 低 | 大 | 中 |
| 用户深度访谈 | 高 | 小（5-10 人） | 极高 |
| 行为分析（隐式） | 中 | 大 | 高 |
| A/B 盲测 | 中 | 中 | 高 |

**推荐：弹窗问卷（3 个问题，10 秒完成）**

```
Q1: 和之前相比？好多了 / 好一点 / 差不多 / 差一点 / 差多了
Q2: 最明显的感觉？更准确 / 更详细 / 更贴心 / 变慢了 / 变啰嗦了 / 风格变了
Q3: 接下来会继续使用吗？更愿意 / 一样 / 不太想用
```

**隐式行为信号：**

| 行为 | 可能表示 | 指标 |
|------|---------|------|
| 频繁重新生成 | 不满意 | 重新生成率 ↑ |
| 提前结束对话 | 失去兴趣 | 放弃率 ↑ |
| 复制/粘贴结果 | 满意（想保留） | 复制率 → |
| 返回修改问题 | 回答偏离 | 修改率 ↑ |

### 用户感知变化量化模型

**PSC（Perceived Change Score）= 正面感知比例 - 负面感知比例**

| PSC 值 | 解读 | 行动 |
|--------|------|------|
| > +30% | 绝大多数用户感觉变好了 | ✅ 正常 |
| +10% ~ +30% | 多数感觉变好了 | ✅ 关注负向 |
| -10% ~ +10% | 感知不一致或没变化 | ⚠️ 需关注 |
| < -10% | 普遍感觉变差了 | 🔴 考虑回滚 |

## 5. 回滚机制

### 触发条件

| 级别 | 条件 | 响应 | 回滚方式 |
|------|------|------|---------|
| P0 | 安全/合规违规 | 立即 | 全量回滚 |
| P0 | 核心指标下降 > 20% | 立即 | 全量回滚 |
| P1 | 负面反馈率 > 5% | 30 分钟 | 按用户回滚 |
| P1 | 错误率 > 1% | 30 分钟 | 全量回滚 |
| P2 | 非核心指标持续下降 | 观察 24h | 按需 |

### 回滚方案

| 方案 | 实现 | 速度 | 影响 | 适用 |
|------|------|------|------|------|
| 全量回滚 | 切换 DNS/流量到旧模型 | < 5min | 全部用户 | P0 |
| 按用户回滚 | 撤回指定用户组 | < 10min | 部分用户 | P1 |
| 自动回滚 | 监控自动触发 | < 2min | 全部用户 | 已配置 |
| 降级回滚 | 保留新模型，回滚部分功能 | < 30min | 部分功能 | 局部问题 |

### 全量回滚执行流程

```
触发回滚（自动/手动）
→ 步骤 1：切换流量到旧模型（修改负载均衡/DNS）
→ 步骤 2：确认旧模型正常响应
→ 步骤 3：更新灰度状态为"已回滚"
→ 确认用户已用旧模型
→ 汇总影响数据，通知团队
→ 开始根因分析
```

### 回滚沟通模板

```markdown
## 模型回滚通知
级别：P0/P1/P2
回滚时间：2026-05-14 14:30
版本：模型 B → 模型 A
原因：[描述]
影响：约 X 名用户，Y 分钟
状态：✅ 已恢复

后续：
- [ ] 根因分析报告（Owner, DDL）
- [ ] 修复方案（Owner, DDL）
- [ ] 重发计划（Owner, DDL）
```

### 回滚复盘

```
1. 事件概述：什么问题？多少用户影响？
2. 根因分析：为什么发生？为什么测试没发现？
3. 流程改进：
   | 漏洞 | 改进措施 | Owner | ETA |
   |------|---------|-------|-----|
   | 测试未覆盖场景 X | 补充用例 | [姓名] | [日期] |
   | 放量太快 | 延长 5%→20% 观察期 | [姓名] | [日期] |
4. 行动项：修复、更新测试集、更新灰度流程
```

## 6. A/B 测试设计

### 离线+在线双轨体系

```
离线评估（Pre-release）： 
  数据：评估数据集（500-2000 条）
  指标：准确率、格式合规、幻觉率
  频率：每次模型更新

在线评估（In-production）：
  数据：真实用户交互数据
  指标：任务完成率、满意度、留存
  频率：灰度期间持续
```

### 实验设计 Checklist

- [ ] 明确假设：H0 无差异，H1 新模型优于旧模型
- [ ] 确定主要指标（如任务完成率）、次要指标、负向指标
- [ ] 计算样本量：预期效果 ≥ 5%，功效 80%，α=0.05，每组 ≥ 1000 session
- [ ] 确定时长：最少 3 天，覆盖工作日和周末
- [ ] 分桶方式：随机按用户 ID hash，组间无污染
- [ ] 分析方式：频率派 p 值或贝叶斯方法

### 常见陷阱

**陷阱 1：辛普森悖论**
整体数据实验组更好，但分群后每群都是对照组更好
→ 预防：做分层分析（segmentation analysis）

**陷阱 2：新奇效应**
用户因"新版本"本身感到新鲜而给好评，但不代表长期
→ 预防：实验持续足够长时间（至少 1-2 周）

**陷阱 3：新模型看起来更好但实际更差**
模型回答更长、更详细 → 人工评分更高 → 但用户更喜欢简洁
→ 预防：盲评时隐藏版本信息，多维度评估，结合行为数据

**陷阱 4：指标矛盾**
准确率提升但用户满意度下降
→ 预防：定义主要指标时考虑 trade-off，设定多维度门槛

### 避免陷阱的最佳实践

| 实践 | 说明 |
|------|------|
| 预注册实验设计 | 实验开始前固定分析方案，防止 P-hacking |
| 分层分析 | 按用户群/场景分层 |
| 足够时长 | 覆盖 1 个完整使用周期 |
| 多维度指标 | 质量 + 行为 + 业务指标 |
| 盲评设计 | 标注员不知道模型版本 |
| 结果可重现 | 相同分析方法得相同结果 |

## 7. 迭代节奏与版本管理

### 升级频率

| 产品阶段 | 升级频率 | 灰度周期 |
|---------|---------|---------|
| 快速探索期 | 每周 | 1-3 天 |
| 成长期 | 每两周 | 3-7 天 |
| 成熟期 | 每月 | 7-14 天 |
| 维护期 | 按需 | 按需 |

### 版本命名

```
v{major}.{minor}.{patch}-{model_info}
示例：v2.1.0-gpt4o-0625

Major：模型架构变更（GPT-3.5 → GPT-4）
Minor：模型版本升级（GPT-4-0314 → GPT-4-0613）
Patch：Prompt 优化/小调整
```

### 持续迭代机制

1. **月度模型 Review**：检查线上表现，评估升级必要性，规划下次升级
2. **每次升级复盘**：灰度数据回顾，用户反馈 Review，流程优化
3. **技术跟踪**：跟踪新模型发布，收集用户对能力的诉求，评估新技术成熟度
