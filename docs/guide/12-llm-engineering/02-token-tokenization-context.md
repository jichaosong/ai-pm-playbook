# 第 2 课：Token、Tokenization 和上下文窗口

> 本课目标：理解 LLM 为什么要把文字切成 token，token 如何影响成本、速度、上下文窗口、Prompt 设计、RAG 和微调。

---

## 1. 什么是 token？

简单说：

> token 是模型读文字时用的小块。

这个“小块”可能是：

- 一个中文字
- 一个英文单词
- 一个英文单词的一部分
- 一个标点符号
- 一个空格
- 一段常见字符组合

例如英文：

```text
I love artificial intelligence.
```

可能会被切成类似：

```text
I | love | artificial | intelligence | .
```

但也可能是：

```text
I | love | artificial | intellig | ence | .
```

这取决于模型使用的 tokenizer。

**Tokenizer** 就是“切词器”。

---

## 2. 中文 token 长什么样？

中文和英文不太一样。

中文没有天然空格，所以 tokenizer 可能这样切：

```text
我喜欢人工智能
```

可能变成：

```text
我 | 喜欢 | 人工 | 智能
```

也可能变成：

```text
我 | 喜 | 欢 | 人 | 工 | 智 | 能
```

不同模型切法不同。

你现在不需要记住具体规则，只要记住：

> token 是模型真正看到的文字单位。

---

## 3. Tokenization 是什么？

**Tokenization** 就是：

> 把人类文字切成 token 的过程。

比如：

```text
你好，世界！
```

经过 tokenization 后，可能变成：

```text
你 | 好 | ， | 世界 | ！
```

再往后，这些 token 会被转成数字 ID。

例如：

```text
你 → 1234
好 → 5678
， → 42
世界 → 91011
！ → 99
```

模型真正处理的是这些数字。

所以完整流程是：

```text
人类文字
→ 切成 token
→ token 变成数字 ID
→ 输入模型
```

---

## 4. 生活类比：切菜

你可以把 tokenization 想成做饭前的切菜。

你要炒菜，不能把一整根胡萝卜直接丢进锅里。

你需要先切成片、丝、块。

LLM 也一样。

它不能直接吃整篇文章。

它需要先把文章切成小块。

这些小块就是 token。

> Tokenization 就是给 AI “切菜”。

---

## 5. 为什么 token 很重要？

因为很多 AI 系统都是按 token 计算的。

它影响：

- 成本
- 速度
- 上下文长度
- 记忆范围
- 输出长度
- 推理速度
- 微调数据大小

很多 API 的计费方式常见是：

```text
输入 token 数 + 输出 token 数
```

输入越长，费用越高。

输出越长，费用也越高。

---

## 6. 什么是上下文窗口？

**上下文窗口**，英文是 **context window**。

简单说：

> 模型一次最多能看到多少 token。

比如一个模型的上下文窗口是：

```text
8K tokens
```

意思是：

> 它一次最多能处理大约 8000 个 token。

如果是：

```text
128K tokens
```

意思是：

> 它一次可以处理更长的文章、更多聊天记录、更多代码。

但注意：

> 上下文窗口不是永久记忆。

它只是“这一次对话中模型眼前能看到的内容”。

---

## 7. 生活类比：桌面大小

把上下文窗口想成你的书桌。

小书桌只能同时放：

- 一本书
- 一支笔
- 一张纸

大书桌可以同时放：

- 十本书
- 电脑
- 笔记本
- 多份资料

LLM 的上下文窗口也是这样。

上下文窗口越大，它一次能“摊开看”的资料越多。

但如果资料太多，超过桌面大小，就放不下。

---

## 8. Token 和上下文窗口的关系

假设模型上下文窗口是：

```text
4096 tokens
```

你输入了：

```text
3000 tokens
```

那它最多还能输出大约：

```text
1096 tokens
```

因为：

```text
输入 + 输出 ≤ 上下文窗口
```

所以如果你把太长的文档直接塞给模型，可能会遇到：

- 内容被截断
- 回答不完整
- 模型忘记前文
- 生成空间不够

---

## 9. 为什么中文、英文 token 数不同？

同样长度的意思，不同语言的 token 数可能不一样。

英文通常按单词或词片段切。

中文可能按字、词或词片段切。

例如：

```text
I love AI.
```

可能很少 token。

中文：

```text
我喜欢人工智能。
```

也可能是几个 token，也可能更多。

这取决于 tokenizer 的训练方式。

对产品经理来说，重点不是记住精确数字，而是理解：

> 不同语言、不同模型，token 成本可能不同。

---

## 10. Token 会影响 Prompt 设计

假设你写了一个很长的 prompt：

```text
你是一个非常专业、非常负责、非常严谨、非常有耐心、非常细致、非常……
```

这会占用很多 token。

有时候你可以简化成：

```text
你是一个严谨、耐心的 AI 教师。
```

意思差不多，但更省 token。

好的 prompt 不一定越长越好。

好的 prompt 是：

> 信息清楚，结构合理，不浪费 token。

---

## 11. Token 会影响 RAG

RAG 是让 AI 先从资料库里找相关资料，再回答。

问题是：

> 找出来的资料也要占 token。

如果你一次塞太多资料进去，模型会读得慢、贵，甚至忽略重点。

所以做 RAG 时要学会：

- 文档切块
- 控制 chunk 大小
- 只取最相关内容
- 去掉重复内容
- 压缩上下文

这些都和 token 有关。

---

## 12. Token 会影响微调

微调时，训练数据也会被切成 token。

训练成本常常和 token 数有关。

如果你的训练数据里有很多废话，例如：

```text
您好您好您好您好您好您好……
```

或者大量重复模板，就会浪费训练成本。

所以微调前要做数据清洗。

这也是为什么后面要讲：

- dataset cleaning
- dataset formatting
- data curation

---

## 13. 小代码示例：粗略估算 token

不同模型有不同 tokenizer。

但初学时可以先粗略估算：

```python
text = "我想学习大语言模型微调。"

# 非严格估算：中文可以先粗略按字数估
rough_tokens = len(text)

print(rough_tokens)
```

这不是精确结果。

只是帮助你先建立直觉：

> 文本越长，token 越多，成本越高，占用上下文越多。

以后真正做工程时，要用对应模型的 tokenizer 来算。

---

## 14. Token、字、词的区别

| 概念 | 简单解释 | 例子 |
|---|---|---|
| 字符 | 最小文字单位 | 我、A、！ |
| 词 | 人类理解的词语 | 人工智能、模型 |
| token | 模型使用的小块 | 可能是字、词、词的一部分 |
| token ID | token 对应的数字 | “AI” → 12345 |

一句话：

> 人看词，模型看 token，电脑算数字。

---

## 15. 对产品经理有什么用？

你做 AI 产品时，token 很关键。

它会影响：

### 成本设计

- 用户一次最多上传多少字？
- 每次对话允许多长？
- 免费用户限制多少 token？
- 高级用户给多少上下文？

### 体验设计

- 回答太慢怎么办？
- 长文档怎么处理？
- 聊天历史太长怎么办？
- 是否需要摘要记忆？

### 技术方案

- 用短上下文模型还是长上下文模型？
- 用 RAG 还是直接塞全文？
- 是否需要压缩 prompt？
- 是否需要缓存系统？

---

## 16. 本课总结

Token 是模型读取文字的小块。

Tokenization 是把文字切成 token 的过程。

模型不是直接处理文字，而是：

```text
文字 → token → 数字 → 模型计算 → 输出 token → 文字
```

Token 影响：

- 成本
- 速度
- 上下文长度
- prompt 设计
- RAG 效果
- 微调成本
- 部署体验

---

## 17. 心智模型

把 token 想成：

> AI 吃进去的一小口食物。

文章太长，就是食物太多。

上下文窗口就是 AI 的盘子。

盘子越大，一次能装的食物越多。

但食物太多也会：

- 吃得慢
- 花钱多
- 消化不好
- 忽略重点

---

## 18. 初学者常见错误

### 错误 1：以为 token 等于字

不一定。

token 可能是一个字，也可能是一个词，也可能是词的一部分。

### 错误 2：以为上下文窗口等于记忆

上下文窗口只是模型当前能看到的内容。

它不是永久记忆。

### 错误 3：prompt 越长越好

不一定。

长 prompt 可能更贵、更慢，还可能让重点变模糊。

### 错误 4：把全文都塞给模型

很多时候应该先筛选、摘要、切块，而不是把所有内容都丢进去。

## 19. 小练习

拿下面这句话：

```text
我想做一个 AI 客服机器人，帮助用户查询订单和退款。
```

试着手动切 token。

你可以先粗略切成：

```text
我 | 想 | 做 | 一个 | AI | 客服 | 机器人 | ， | 帮助 | 用户 | 查询 | 订单 | 和 | 退款 | 。
```

然后思考：

> 如果我要让模型回答得更好，哪些 token 是最重要的？

答案通常是：

```text
AI 客服机器人
订单
退款
用户
查询
```

这些是语义重点。

---

## 20. 小项目

做一个简单的“prompt 压缩练习”。

原 prompt：

```text
你是一个非常优秀、非常耐心、非常专业、非常懂人工智能、非常擅长解释复杂概念的老师。请你用非常简单、非常容易理解、非常适合初学者的语言，帮我解释什么是 token 和 tokenization。
```

压缩成：

```text
你是耐心的 AI 老师。用初学者能懂的中文解释 token 和 tokenization。
```

对比两个输出。

你会学到：

> 好 prompt 不是堆很多形容词，而是清楚表达任务、对象和输出要求。
