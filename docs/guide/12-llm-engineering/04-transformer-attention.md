# 第 4 课：Transformer 和 Attention

> 本课目标：用简单语言理解 Transformer 和 Attention。你不需要先懂复杂数学，只要先建立一个清楚的工程直觉：模型为什么能看懂一句话里不同位置之间的关系。

---

## 1. 先说结论

现代 LLM 之所以强，很大一部分原因来自一种模型结构：

> Transformer。

Transformer 里最重要的核心能力叫：

> Attention，注意力机制。

简单说：

> Attention 让模型在处理一个 token 时，可以看看其他 token，判断哪些内容最重要。

比如这句话：

```text
小明把苹果放进书包，因为它太重了。
```

这里的“它”指什么？

可能是“书包”，也可能是“苹果”，要看上下文。

Attention 的作用就是帮助模型判断：

> 当前这个词，应该重点关注前面哪些词？

---

## 2. 生活类比：开会时抓重点

想象你在开一个产品评审会。

会议里有很多人说话：

- 用户研究说：用户觉得流程太复杂。
- 设计师说：页面按钮太多。
- 工程师说：接口响应慢。
- 老板说：这个月必须提升转化率。

你在做决策时，不会把每句话都当成同等重要。

如果你现在要优化转化率，你会重点关注：

- 用户觉得流程太复杂
- 页面按钮太多
- 老板要求提升转化率

如果你现在要优化性能，你会重点关注：

- 接口响应慢

这就是 attention 的感觉：

> 面对很多信息时，根据当前任务，决定该重点看什么。

---

## 3. 为什么旧方法不够好？

在 Transformer 流行之前，处理文本常用一些按顺序读的模型。

你可以把它想成：

> 一个人从左到右读句子，一边读一边记。

这种方法有一个问题：

如果句子很长，前面的信息容易被忘掉。

例如：

```text
小王昨天在公司开会时提到，虽然项目延期、预算紧张、客户临时改需求，但他依然认为最重要的问题是用户注册流程太复杂。
```

最后的重点是：

```text
用户注册流程太复杂
```

但句子很长，中间夹了很多信息。

旧方法可能更容易丢掉远处的信息。

Transformer 的优势是：

> 它可以让句子里的每个 token 直接关注其他 token，不一定只能一步一步传话。

---

## 4. Attention 到底在做什么？

我们用很简单的话说。

当模型看到一个 token 时，它会问：

> 为了理解我当前这个 token，我应该看前面哪些 token？它们分别有多重要？

比如：

```text
我把手机放在桌子上，因为它没电了。
```

当模型看到“它”时，它可能会更关注：

```text
手机
```

而不是：

```text
桌子
```

因为“没电了”更像是在描述手机。

所以 attention 会给不同 token 分配不同“注意力权重”。

你可以理解成：

```text
手机：90% 重要
桌子：10% 重要
```

真实模型不是这么简单，但直觉就是这样。

---

## 5. 一个更产品化的例子

用户问：

```text
帮我总结这份 PRD，重点关注用户权限和计费策略。
```

模型看到“重点关注”时，会更注意：

- 用户权限
- 计费策略
- PRD 中相关段落

它不会平均关注整份文档里的所有内容。

这就是为什么好的 prompt 能影响模型输出。

因为 prompt 会告诉模型：

> 哪些信息更重要。

---

## 6. Self-Attention 是什么？

你会经常看到一个词：

> Self-Attention，自注意力。

简单说：

> Self-Attention 是一句话内部的 token 互相看彼此。

比如这句话：

```text
这个产品虽然功能很多，但用户觉得很难用。
```

模型在理解“难用”时，会回头看：

- 产品
- 功能很多
- 用户觉得

它在理解“虽然”时，会知道后面可能有转折。

所以 self-attention 不是一个词孤零零地理解自己，而是：

> 每个词都可以参考其他词，再形成更好的理解。

---

## 7. Multi-Head Attention 是什么？

你还会看到：

> Multi-Head Attention，多头注意力。

“多头”听起来很吓人，其实可以这样理解：

> 模型不只用一种视角看句子，而是用多种视角同时看。

像产品评审一样。

同一段用户反馈：

```text
用户觉得注册流程太长，而且验证码经常收不到，最后很多人直接放弃。
```

产品经理可能关注：

- 转化率
- 用户体验
- 注册流程

工程师可能关注：

- 验证码服务
- 接口稳定性

增长同学可能关注：

- 流失点
- 漏斗数据

Multi-head attention 就像模型内部有多个观察角度。

不同 head 可能关注不同关系：

- 谁是主语
- 谁在修饰谁
- 哪些词表示因果
- 哪些词表示转折
- 哪些内容和任务目标相关

---

## 8. Transformer 的整体结构怎么想？

不要一开始记复杂结构图。

先记这个简化流程：

```text
输入 token
→ 变成 embedding
→ 加上位置信息
→ 经过很多层 Transformer block
→ 每层都做 attention 和信息加工
→ 输出下一个 token 的概率
```

每一层 Transformer block 可以粗略理解成一个加工车间。

每个车间会做两类事：

1. 看上下文里哪些 token 重要。
2. 对信息做进一步加工。

很多层叠起来后，模型就能从简单模式学到复杂模式。

---

## 9. 为什么需要位置信息？

Attention 很强，但它本身有一个问题：

> 如果只看 token 内容，不知道顺序，句子意思会乱。

例如：

```text
狗咬人
```

和：

```text
人咬狗
```

这两个句子 token 差不多，但意思完全不同。

所以模型需要知道每个 token 的位置。

这叫：

> positional encoding，位置编码。

简单说：

> 给每个 token 加上“我在第几个位置”的信息。

就像排队时每个人有号码牌。

如果没有号码牌，大家站乱了，意思也乱了。

---

## 10. Attention 为什么适合长文本？

Attention 的优势是：

> 远处的词也可以被直接关注。

例如：

```text
这款产品在去年上线时获得了很多好评，虽然早期版本存在一些性能问题，后续团队通过重构架构和优化缓存解决了大部分问题，因此它现在适合企业客户使用。
```

理解“它”时，模型需要知道“它”指的是“这款产品”。

这两个位置隔得比较远。

Attention 可以帮助模型跨距离建立关系。

这也是 LLM 能处理长文档、长代码、长对话的重要原因之一。

但注意：

> 上下文变长不代表模型一定能完美理解所有内容。

长上下文仍然会带来：

- 成本变高
- 速度变慢
- 重点稀释
- 检索困难
- 幻觉风险

所以工程上仍然需要 RAG、摘要、chunking、rerank 等方法。

---

## 11. Attention 和 Prompt 有什么关系？

Prompt 的本质之一，是告诉模型：

> 哪些信息更重要，应该用什么方式处理。

比如你写：

```text
请总结这篇文章。
```

模型会自己判断重点。

如果你写：

```text
请从产品经理角度总结这篇文章，重点关注用户痛点、核心功能、商业模式和上线风险。
```

模型的 attention 会更容易被这些目标牵引。

所以好的 prompt 不是“魔法咒语”，而是在帮助模型分配注意力。

---

## 12. Attention 和 RAG 有什么关系？

RAG 会把检索到的文档片段放进 prompt。

但模型不一定会平均使用所有片段。

它会根据问题和上下文，决定哪些片段更重要。

所以 RAG 设计里有一个关键问题：

> 你放进上下文的内容，是否真的值得模型注意？

如果塞进去很多无关内容，模型可能：

- 看错重点
- 忽略关键证据
- 生成含糊答案
- 把无关资料混在一起

所以好的 RAG 不只是“多找资料”，而是：

> 找准资料，控制数量，组织清楚，让模型容易注意到关键证据。

---

## 13. Attention 和幻觉有什么关系？

幻觉不是单靠 attention 就能解决的。

但 attention 设计会影响模型如何使用上下文。

如果上下文里没有答案，模型可能根据训练中学到的模式编一个看似合理的回答。

如果上下文里有很多冲突信息，模型也可能选错重点。

比如知识库里同时有：

```text
旧规则：订单完成后 30 天内可退款。
新规则：订单完成后 7 天内可退款。
```

如果没有版本管理，模型可能引用旧规则。

这不是单纯的模型问题，也可能是产品和数据工程问题。

---

## 14. Transformer 为什么适合并行计算？

旧式顺序模型像一个人排队传话：

```text
第 1 个词 → 第 2 个词 → 第 3 个词 → 第 4 个词
```

很多步骤必须按顺序来。

Transformer 训练时更像大家同时开会：

```text
很多 token 可以同时互相看
```

这更适合 GPU 并行计算。

GPU 擅长同时做大量类似的数学计算。

所以 Transformer 不只是效果好，也更适合大规模训练。

这也是现代 LLM 能做大的重要原因之一。

---

## 15. 但 Attention 也有成本

Attention 很强，但不是免费的。

传统 attention 在长上下文时计算量会明显增加。

直觉上：

> token 越多，每个 token 要看的其他 token 越多，计算就越重。

比如 100 个 token 之间互相关注，关系数量还可以接受。

如果是 10000 个 token，关系数量就会大很多。

所以工程上会有很多优化技术，比如：

- KV cache
- Flash Attention
- sliding window attention
- grouped-query attention
- speculative decoding

这些我们后面会讲。

现在先记住：

> 长上下文很有用，但会带来速度和成本问题。

---

## 16. 一个简单的注意力例子

句子：

```text
小红把笔记本电脑放进包里，因为它很贵。
```

模型理解“它”时，可能更关注：

```text
笔记本电脑
```

因为“很贵”更像描述电脑。

如果句子变成：

```text
小红把笔记本电脑放进包里，因为它很大。
```

“它”可能更像指“笔记本电脑”，也可能根据上下文考虑“包”。

如果句子是：

```text
小红把笔记本电脑放进包里，因为它能保护电脑。
```

“它”更可能指“包”。

同一个“它”，根据后面信息不同，attention 关注对象也可能不同。

---

## 17. 小代码示例：模拟 attention 思想

下面不是 Transformer 的真实代码，只是帮助理解“不同词有不同重要性”。

```python
words = ["用户", "觉得", "注册流程", "太长", "导致", "放弃"]

# 假设我们现在要分析“流失原因”
attention_scores = {
    "用户": 0.1,
    "觉得": 0.1,
    "注册流程": 0.3,
    "太长": 0.25,
    "导致": 0.1,
    "放弃": 0.15,
}

for word in words:
    print(word, attention_scores[word])
```

这个例子表达的思想是：

> 同一句话里，不同词对当前任务的重要性不同。

真实 attention 是模型自己通过训练学出来的，不是我们手写分数。

## 18. Transformer、Attention、Embedding 的关系

我们把前几课串起来：

| 概念 | 简单解释 | 类比 |
|---|---|---|
| Token | 模型读文字的小块 | 一口食物 |
| Embedding | token 的意思坐标 | 地图位置 |
| Attention | 判断哪些 token 更重要 | 开会抓重点 |
| Transformer | 一层层处理 token 的模型结构 | 信息加工工厂 |

完整流程可以这样想：

```text
文字
→ 切成 token
→ token 变成 embedding
→ 加上位置信息
→ Transformer 用 attention 看上下文重点
→ 预测下一个 token
→ 生成回答
```

## 19. 对产品经理有什么用？

产品经理不需要手写 Transformer，但需要知道它带来的产品影响。

### 影响 1：Prompt 要有重点

不要把所有要求堆在一起。

更好的做法是结构化：

```text
目标：总结 PRD
重点：用户痛点、核心功能、上线风险
输出：用表格
限制：不要编造文档中没有的信息
```

这能帮助模型更好地分配注意力。

### 影响 2：RAG 不要塞太多无关内容

上下文不是垃圾桶。

放进去的资料越多，不一定越好。

### 影响 3：长上下文不是万能药

长上下文可以让模型看到更多内容，但也可能让重点变弱。

很多时候，好的检索和摘要比盲目拉长上下文更重要。

### 影响 4：评估要看“是否抓对重点”

AI 输出质量不仅是语言是否流畅，还要看：

- 有没有抓住关键证据
- 有没有忽略重要约束
- 有没有引用错误段落
- 有没有混淆旧规则和新规则

这些都和模型如何使用上下文有关。

---

## 20. 本课总结

Transformer 是现代 LLM 的核心结构之一。

Attention 是 Transformer 里的关键能力。

Attention 让模型在处理一个 token 时，可以关注上下文中更重要的 token。

它让模型更擅长理解：

- 指代关系
- 上下文关系
- 因果关系
- 转折关系
- 长距离依赖
- 任务重点

但 attention 也会带来计算成本，特别是在长上下文场景中。

---

## 21. 心智模型

把 Transformer 想成：

> 一个很大的信息加工工厂。

把 Attention 想成：

> 工厂里的重点筛选员。

每来一段话，它都会问：

```text
现在这个词要理解清楚，应该重点看哪些词？
```

然后一层层加工，最后生成回答。

---

## 22. 初学者常见错误

### 错误 1：以为 attention 等于人类注意力

不是。

它只是一个数学机制，用来计算 token 之间的相关程度。

它像人类注意力，但不等于人类意识。

### 错误 2：以为上下文越长越好

不一定。

上下文越长，成本越高，重点也可能越分散。

### 错误 3：以为模型会自动关注你想让它关注的内容

不一定。

你需要通过 prompt、结构、标题、引用、排序等方式帮助模型抓重点。

### 错误 4：以为 RAG 检索到内容就一定会被正确使用

不一定。

模型可能忽略关键段落，也可能被无关段落干扰。

## 23. 小练习

看下面这句话：

```text
用户取消订单不是因为价格贵，而是因为配送时间太长。
```

如果模型要回答：

```text
用户取消订单的主要原因是什么？
```

它应该重点关注哪些词？

建议答案：

```text
不是因为价格贵
而是因为配送时间太长
```

重点是转折关系。

如果忽略“不是……而是……”，模型可能错误回答“价格贵”。

## 24. 小项目

做一个“让模型抓重点”的 prompt 实验。

准备一段文本：

```text
这款 AI 客服产品上线后，用户咨询量下降了 20%，但人工客服满意度也下降了。主要原因是 AI 能回答常见问题，但在退款、投诉、异常订单等复杂场景中经常答非所问。用户最不满意的是无法快速转人工。
```

先问：

```text
总结这段话。
```

再问：

```text
从 AI 产品经理角度总结这段话，重点关注：
1. 正向效果
2. 主要风险
3. 用户最不满意的问题
4. 下一步产品改进建议
```

比较两个回答。

你会发现：

> 明确重点后，模型更容易关注关键内容。
