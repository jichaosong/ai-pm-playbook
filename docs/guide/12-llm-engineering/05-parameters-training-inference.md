# 第 5 课：参数、训练和推理

> 本课目标：理解模型里的“参数”是什么，训练和推理有什么区别，以及这些概念如何影响 AI 产品的成本、速度、质量和上线方案。

---

## 1. 先说结论

LLM 里有很多很多**参数**。

简单说：

> 参数是模型内部用来做判断的小数字。

训练时，模型会不断调整这些数字。

推理时，模型会使用这些已经调好的数字来生成回答。

所以：

```text
训练 = 调整模型
推理 = 使用模型
```

这是理解 LLM 工程的核心区别。

---

## 2. 什么是参数？

参数听起来很抽象。

你可以先把它想成：

> 模型内部很多很多可以调节的小旋钮。

模型不是靠一条条人工写好的规则工作。

它靠大量参数共同决定：

- 哪个 token 更可能出现
- 哪些词之间有关联
- 哪种回答风格更合适
- 哪段上下文更重要
- 代码应该怎么补全
- 问题应该怎么回答

比如模型看到：

```text
中国的首都是
```

它会通过参数计算出下一个 token 更可能是：

```text
北京
```

而不是：

```text
香蕉
```

---

## 3. 生活类比：调音台

想象一个调音师面前有一个巨大的调音台。

上面有很多旋钮：

- 音量
- 低音
- 高音
- 混响
- 人声
- 鼓声
- 吉他声

调音师不断调这些旋钮，让音乐听起来更好。

LLM 的参数也像这些旋钮。

训练模型时，就像不断调旋钮。

目标是让模型越来越会预测正确的下一个 token。

---

## 4. 参数越多越好吗？

不一定。

参数多通常意味着模型容量更大。

“容量”可以理解成：

> 模型能装下和表达更多复杂模式的能力。

比如：

```text
1B 参数 = 10 亿参数
7B 参数 = 70 亿参数
70B 参数 = 700 亿参数
```

大模型通常更强，尤其在：

- 复杂推理
- 多语言理解
- 长文档处理
- 代码生成
- 多任务能力

但大模型也通常：

- 更慢
- 更贵
- 更吃显存
- 更难部署
- 推理成本更高

所以工程上不是永远选最大模型。

产品上更常见的问题是：

> 用多大的模型，刚好够解决当前任务？

---

## 5. 小模型什么时候更好？

小模型在很多场景反而更合适。

比如：

- 分类任务
- 简单客服问答
- 固定格式抽取
- 本地设备运行
- 高并发低成本场景
- 对响应速度要求很高的场景

例如一个任务只是判断用户反馈是：

```text
投诉 / 建议 / 咨询 / 表扬
```

这不一定需要超大模型。

一个小模型或者传统机器学习方法可能已经够用。

产品经理要记住：

> 模型选择不是炫技，而是成本、质量、速度和风险的平衡。

---

## 6. 什么是训练？

训练就是：

> 用大量数据调整模型参数，让模型更会预测下一个 token。

训练过程可以粗略理解为：

```text
给模型一句话
→ 让它猜下一个 token
→ 看它猜得对不对
→ 如果错了，就调整参数
→ 重复很多很多次
```

比如训练数据里有：

```text
天空是蓝色的。
```

模型看到：

```text
天空是
```

它可能一开始猜错。

训练系统会告诉它：

```text
正确答案更接近“蓝色”
```

然后参数被微调一点点。

重复海量数据后，模型慢慢学到很多语言和知识模式。

---

## 7. 生活类比：练习考试

训练模型像学生刷题。

学生做题：

1. 看题目
2. 写答案
3. 对答案
4. 知道哪里错了
5. 改进方法
6. 继续做下一题

模型训练也是类似。

只不过学生改的是自己的理解方法，模型改的是内部参数。

---

## 8. 什么是推理？

推理，英文是 inference。

简单说：

> 推理就是使用训练好的模型来生成回答。

当你在 ChatGPT、Ollama、Claude、Gemini 或其他 AI 产品里输入问题时，模型通常正在做推理。

推理流程大概是：

```text
用户输入
→ 切成 token
→ 模型用参数计算
→ 预测下一个 token
→ 不断生成
→ 输出回答
```

推理时，模型参数通常不会改变。

它只是使用已经学好的参数。

---

## 9. 训练 vs 推理

| 对比项 | 训练 | 推理 |
|---|---|---|
| 目的 | 调整模型能力 | 使用模型能力 |
| 参数会不会变 | 会变 | 通常不变 |
| 成本 | 很高 | 相对低，但大量调用也很贵 |
| 数据 | 大量训练数据 | 用户输入和上下文 |
| 时间 | 可能数小时、数天、数周 | 通常几百毫秒到几十秒 |
| 类比 | 学生学习 | 学生考试答题 |
| 产品阶段 | 模型研发 / 微调 | 用户真实使用 |

一句话记住：

> 训练是在“学”，推理是在“用”。

---

## 10. 什么是预训练？

预训练，英文是 pretraining。

简单说：

> 预训练是模型第一次大规模学习语言和世界知识的阶段。

模型会读大量文本，例如：

- 网页
- 书籍
- 代码
- 百科
- 论文
- 对话
- 文档

它学会：

- 语言规律
- 基础知识
- 代码结构
- 常见表达
- 推理模式

预训练成本非常高。

普通团队通常不会从零预训练一个大模型。

更多时候会使用已有开源模型或闭源模型。

---

## 11. 什么是微调？

微调，英文是 fine-tuning。

简单说：

> 微调是在已有模型基础上，用较小的数据继续训练，让它更适合某个任务或风格。

比如一个通用模型本来什么都懂一点。

你可以用客服数据微调它，让它更像客服助手。

也可以用法律文档微调它，让它更熟悉法律表达。

也可以用代码评审数据微调它，让它更擅长 code review。

微调不是重新造一个模型。

它更像：

> 给已经毕业的人做岗位培训。

---

## 12. 什么是继续预训练？

继续预训练，英文常叫 continued pretraining。

它和普通微调不完全一样。

简单理解：

> 继续预训练是让模型继续阅读某个领域的大量原始文本，补充领域知识和语言习惯。

比如你有很多医疗文档、金融报告、法律材料、代码仓库。

你可以让模型继续在这些文本上学习。

它更像：

> 让模型继续读专业书。

而 SFT 更像：

> 教模型如何按指令回答问题。

两者目标不一样。

---

## 13. 什么是 SFT？

SFT 是 Supervised Fine-Tuning。

中文可以叫：

> 监督式微调。

简单说：

> 给模型很多“问题 → 好答案”的样本，让它学会按这种方式回答。

例如：

```text
用户：帮我写一封请假邮件
助手：你好，我因身体不适需要请假一天……
```

SFT 让模型学会：

- 听指令
- 按格式输出
- 使用某种语气
- 遵守回答规范
- 适配业务场景

后面我们会单独讲 SFT。

---

## 14. 参数和知识的关系

很多初学者会问：

> 模型知识是不是都存在参数里？

粗略说，是的，模型在训练中学到的很多模式会被压缩进参数里。

但要小心理解。

参数不是一本可直接翻阅的书。

它更像：

> 大量知识和语言模式被压缩进一个复杂的数字网络。

这也是为什么模型会：

- 知道很多常识
- 但不一定知道最新信息
- 能生成流畅回答
- 但有时会编造细节
- 记得训练中常见模式
- 但不可靠地记住每个事实

所以如果产品需要准确使用企业知识，不能只靠参数。

通常需要 RAG、工具调用或数据库查询。

---

## 15. 训练数据和参数的关系

训练数据会影响参数。

参数会影响模型行为。

所以：

```text
数据质量 → 参数状态 → 模型输出质量
```

如果训练数据里有很多错误、偏见、过期内容，模型也可能学到这些问题。

这就是为什么数据清洗很重要。

坏数据训练出来的模型，就像学生一直看错误答案。

---

## 16. 推理时为什么会有随机性？

你可能发现，同一个问题问两次，模型回答不完全一样。

这是因为推理时模型不是永远选唯一答案。

它会给下一个 token 算出很多可能性。

例如：

```text
今天天气很好，我们去
```

后面可能是：

- 公园
- 散步
- 爬山
- 海边

模型会根据概率选择。

有些参数设置会影响生成结果，比如：

- temperature
- top_p
- max tokens

简单理解：

| 参数 | 作用 | 类比 |
|---|---|---|
| temperature | 控制随机程度 | 保守 vs 有创意 |
| top_p | 控制候选范围 | 只从高概率答案里选 |
| max tokens | 控制最长输出 | 限制回答长度 |

后面讲推理优化时会更深入。

---

## 17. Temperature 是什么？

Temperature 可以理解成：

> 模型回答的冒险程度。

低 temperature：

- 更稳定
- 更保守
- 更适合事实问答、分类、结构化输出

高 temperature：

- 更发散
- 更有创意
- 更适合脑暴、文案、故事生成

例如：

```text
给我 10 个咖啡店名字
```

可以用高一点的 temperature。

但：

```text
从合同中抽取付款日期
```

应该用低一点的 temperature。

产品经理要记住：

> 生成参数也是产品体验的一部分。

---

## 18. 为什么推理会慢？

LLM 生成回答通常是一个 token 一个 token 地生成。

它不是一次性吐出整段话。

流程是：

```text
预测第 1 个 token
→ 把第 1 个 token 加回上下文
→ 预测第 2 个 token
→ 继续
```

所以输出越长，耗时越久。

影响推理速度的因素包括：

- 模型大小
- 输入 token 数
- 输出 token 数
- GPU 性能
- 是否使用 KV cache
- 是否批量推理
- 服务框架是否高效

后面我们会讲 KV cache、vLLM、Flash Attention 等优化方法。

---

## 19. 为什么训练很贵？

训练贵是因为它不只是“跑一次模型”。

训练要反复：

1. 读大量数据
2. 前向计算
3. 计算错误
4. 反向传播
5. 更新参数
6. 重复很多轮

特别是大模型，参数可能有几十亿、几百亿。

每次更新都需要大量计算和显存。

所以大多数产品团队不会从零训练大模型。

更常见的是：

- 调用闭源模型 API
- 使用开源模型本地推理
- 做 LoRA / QLoRA 微调
- 做 RAG
- 做 prompt engineering

---

## 20. 产品上什么时候需要训练或微调？

不是所有 AI 产品都需要微调。

先问这几个问题：

### 问题 1：Prompt 能解决吗？

如果只是格式、语气、步骤不稳定，可能先优化 prompt。

### 问题 2：RAG 能解决吗？

如果问题是模型不知道你的企业知识，通常先考虑 RAG。

### 问题 3：工具调用能解决吗？

如果问题需要查订单、查库存、查账户，应该接工具或数据库。

### 问题 4：微调真的必要吗？

如果你需要模型稳定学会某种风格、格式、任务模式，才考虑微调。

一句话：

> 微调不是第一个选择，通常是 prompt、RAG、工具都不够时的选择。

---

## 21. 常见方案选择

| 问题类型 | 优先方案 |
|---|---|
| 回答格式不好 | Prompt / 输出约束 |
| 不知道公司文档 | RAG |
| 需要实时订单数据 | 工具调用 / 数据库查询 |
| 语气不符合品牌 | Prompt，必要时 SFT |
| 特定任务长期不稳定 | SFT / LoRA |
| 需要领域语言习惯 | Continued pretraining 或领域微调 |
| 推理太贵 | 小模型、缓存、量化、路由 |
| 推理太慢 | 模型服务优化、KV cache、批处理 |

---

## 22. 模型参数会在用户使用时改变吗？

通常不会。

用户和模型对话时，模型参数一般不会因为这一段对话自动更新。

它只是把当前上下文放进模型里，然后生成回答。

如果产品想让 AI “记住用户偏好”，通常不是直接改模型参数，而是用：

- 用户画像
- 记忆系统
- 数据库存储
- RAG 检索
- 长期 memory

所以：

> 记忆系统不等于模型参数自动更新。

这是非常重要的产品认知。

---

## 23. 本课总结

参数是模型内部的大量数字，决定模型如何处理输入和生成输出。

训练是调整参数。

推理是使用参数。

预训练让模型学会基础语言和知识。

微调让模型更适合特定任务或风格。

继续预训练让模型继续吸收某个领域的大量文本。

产品设计中，不要一上来就想微调，要先判断：

- Prompt 是否够用
- RAG 是否更合适
- 工具调用是否必要
- 成本和风险是否可控

---

## 24. 心智模型

把模型想成一个厨师。

- 参数：厨师脑子里的经验和手感
- 训练：厨师练习做菜
- 预训练：厨师学习大量基础菜系
- 微调：厨师接受某家餐厅的岗位培训
- 推理：厨师真正给客人做菜
- Prompt：顾客点菜要求
- RAG：临时递给厨师的菜单和配方
- 工具调用：厨师去仓库查库存或找收银台查订单

---

## 25. 初学者常见错误

### 错误 1：以为参数越多越好

不一定。

大模型更强，但也更慢、更贵、更难部署。

### 错误 2：以为微调能解决所有问题

不一定。

企业知识更新、实时数据查询、权限判断，通常更适合 RAG 或工具调用。

### 错误 3：以为用户聊天会自动改变模型

通常不会。

对话上下文只是临时输入，不等于模型参数被更新。

### 错误 4：混淆训练和推理

训练是在学，推理是在用。

两者成本、流程、目标都不同。

---

## 26. 小练习

判断下面问题更适合 prompt、RAG、工具调用还是微调：

```text
1. AI 回答太啰嗦，希望变简洁。
2. AI 不知道公司最新退款政策。
3. AI 需要查询用户订单状态。
4. AI 总是不能按照固定客服话术回答。
```

参考答案：

```text
1 → Prompt
2 → RAG
3 → 工具调用 / 数据库查询
4 → Prompt，必要时 SFT / LoRA 微调
```

设计一个“模型方案选择表”。

场景：AI 客服机器人。

填写下面表格：

| 需求 | 方案 | 原因 |

| 回答语气要更礼貌 | Prompt / SFT | Prompt 先试，不稳定再微调 |

> 不要把所有问题都交给微调，而是学会选择合适的工程方案。
