# 第 6 课：开源模型 vs 闭源模型

> 本课目标：理解开源模型和闭源模型的区别，知道什么时候应该用 API 模型，什么时候应该用本地或自托管模型，以及产品经理在选型时应该看哪些因素。

---

## 1. 先说结论

LLM 大致可以分成两类：

```text
闭源模型：你通过 API 或产品使用，通常看不到模型权重
开源模型：你可以下载或自托管，通常能更自由地部署和改造
```

注意，这里的“开源”在 AI 领域有时比较复杂。

有些模型开放权重，但不一定完全开放训练数据、训练代码或商业使用权。

所以更准确地说：

> 选模型时，不要只看“开源”两个字，要看许可证、权重、部署方式、成本、能力和风险。

---

## 2. 什么是闭源模型？

闭源模型通常指：

> 模型由某家公司训练和托管，你通过 API 或网页产品使用它，但看不到完整模型内部。

常见例子包括：

- OpenAI 的模型
- Anthropic Claude
- Google Gemini
- 部分云厂商托管模型

你通常使用方式是：

```text
你的应用
→ 调用模型 API
→ 模型服务返回答案
```

你不需要自己管理 GPU、模型文件、推理框架和扩缩容。

---

## 3. 什么是开源模型？

开源模型通常指：

> 模型权重可以被下载、部署或进一步微调。

常见例子包括：

- Llama 系列
- Qwen 系列
- Mistral 系列
- DeepSeek 系列
- Gemma 系列

你可以把模型运行在：

- 本地电脑
- 公司服务器
- 云 GPU
- 边缘设备
- 手机或笔记本上的轻量环境

使用方式可能是：

```text
下载模型
→ 选择推理框架
→ 部署服务
→ 你的应用调用自己的模型服务
```

---

## 4. 生活类比：外卖 vs 自建厨房

闭源模型像点外卖。

优点：

- 快
- 方便
- 不用买厨房设备
- 不用雇厨师
- 菜品质量通常稳定

缺点：

- 菜价由别人定
- 菜谱你不能改太多
- 高峰期可能排队
- 食材来源你不完全掌控

开源模型像自建厨房。

优点：

- 菜谱可控
- 成本结构可控
- 可以服务内部特殊需求
- 数据不一定要出公司
- 可以深度改造

缺点：

- 要买设备
- 要维护厨房
- 要懂运营
- 出问题要自己修
- 初期门槛更高

一句话：

> 闭源模型更像买服务，开源模型更像建能力。

---

## 5. 闭源模型的优点

### 优点 1：上手快

你只需要申请 API key，然后调用接口。

不需要自己部署模型。

适合：

- 快速 MVP
- 原型验证
- 小团队试错
- 功能上线早期

### 优点 2：综合能力强

闭源大模型通常在通用任务上表现很好，例如：

- 写作
- 总结
- 代码
- 多语言
- 推理
- 工具调用
- 多模态

### 优点 3：运维成本低

你不用管：

- GPU 驱动
- 显存
- 推理框架
- 模型量化
- 服务扩缩容
- 高并发调度

这些由模型服务商处理。

### 优点 4：更新快

模型服务商会持续升级模型。

你可以比较快用到新能力。

---

## 6. 闭源模型的缺点

### 缺点 1：成本可能不可控

API 通常按 token 计费。

如果用户量大、输入长、输出长，成本可能快速上升。

比如：

```text
长文档分析
多轮对话
大量客服请求
批量内容生成
```

都可能烧 token。

### 缺点 2：数据和合规压力

如果业务涉及：

- 医疗数据
- 金融数据
- 企业内部文档
- 用户隐私
- 代码仓库
- 政府或涉密数据

就要认真评估数据是否可以发给第三方 API。

### 缺点 3：可控性有限

你不能完全控制：

- 模型权重
- 训练数据
- 内部推理细节
- 版本升级节奏
- 某些安全策略

### 缺点 4：供应商锁定

如果你的系统深度依赖某个闭源模型特性，将来迁移可能困难。

例如：

- 特定 function calling 格式
- 特定多模态输入方式
- 特定上下文长度
- 特定输出风格

---

## 7. 开源模型的优点

### 优点 1：部署可控

你可以选择：

- 本地部署
- 私有云部署
- 公有云 GPU 部署
- 边缘部署

这对企业安全和合规很重要。

### 优点 2：成本结构更可控

闭源 API 常常是按调用付费。

开源模型部署后，成本更多来自：

- GPU 租赁
- 服务器
- 运维人力
- 存储
- 网络

如果调用量足够大，自托管可能更划算。

### 优点 3：可以微调

开源模型通常更适合做：

- LoRA
- QLoRA
- SFT
- DPO
- 领域微调
- 本地量化

你可以让模型更适合自己的业务。

### 优点 4：可离线运行

某些场景不能依赖外网，例如：

- 工厂现场
- 医疗设备
- 政企内网
- 车载系统
- 安全审计环境

开源或本地模型更适合。

---

## 8. 开源模型的缺点

### 缺点 1：需要工程能力

你需要懂：

- 模型格式
- GPU 显存
- 推理框架
- 量化
- 并发
- 监控
- 模型服务

这对小团队不一定友好。

### 缺点 2：效果不一定比闭源强

开源模型进步很快，但在某些复杂任务上，闭源顶级模型仍可能更强。

尤其是：

- 复杂推理
- 高质量写作
- 多模态理解
- 超长上下文
- 稳定工具调用
- 安全对齐

### 缺点 3：运维责任在你

模型部署后，问题都要自己处理：

- 服务挂了
- GPU 不够
- 推理太慢
- 显存爆了
- 模型输出异常
- 版本升级后效果变差

### 缺点 4：许可证要认真看

不是所有开放模型都能随便商用。

有些模型对：

- 商业使用
- 用户规模
- 再分发
- 训练数据
- 派生模型

有额外限制。

产品经理和法务需要一起确认。

---

## 9. 开源 vs 闭源对比表

| 维度 | 闭源模型 | 开源模型 |
|---|---|---|
| 上手速度 | 快 | 慢一些 |
| 初期成本 | 低 | 可能较高 |
| 大规模调用成本 | 可能变高 | 可能更可控 |
| 数据控制 | 较弱，取决于服务商政策 | 更强，可私有化 |
| 模型可控性 | 较弱 | 较强 |
| 微调自由度 | 通常有限 | 通常更高 |
| 运维压力 | 低 | 高 |
| 能力上限 | 顶级闭源通常很强 | 取决于模型和部署能力 |
| 合规适配 | 需要评估供应商 | 更容易做私有化 |
| 适合阶段 | MVP、快速验证、通用能力 | 成本优化、私有化、深度定制 |

---

## 10. 产品早期怎么选？

如果你在做 0 到 1 产品，通常建议：

> 先用闭源模型快速验证价值，再根据成本、合规和可控性决定是否引入开源模型。

因为早期最重要的是验证：

- 用户是否真的需要这个功能
- AI 是否真的能解决问题
- 交互流程是否成立
- 用户是否愿意付费

如果一开始就自建模型服务，可能会把大量时间花在工程基础设施上，而不是验证产品价值。

---

## 11. 什么时候优先选闭源模型？

以下情况优先考虑闭源模型：

### 场景 1：快速做 MVP

你想尽快验证一个 AI 产品想法。

闭源 API 可以让你快速上线。

### 场景 2：任务复杂，质量要求高

比如：

- 高质量写作
- 复杂推理
- 多语言客服
- 代码生成
- 多模态理解

顶级闭源模型通常更省心。

### 场景 3：团队缺少 AI 基础设施能力

如果团队没有 MLE、平台工程、GPU 运维经验，闭源模型能降低门槛。

### 场景 4：调用量还不大

早期用户量小，API 成本可能可以接受。

---

## 12. 什么时候优先选开源模型？

以下情况优先考虑开源模型：

### 场景 1：数据不能出公司

例如金融、医疗、政企、代码安全等场景。

### 场景 2：调用量很大

如果每天有大量请求，自托管可能更有成本优势。

### 场景 3：需要深度微调

如果你要让模型长期稳定适配某个专业任务，开源模型更灵活。

### 场景 4：需要本地或离线运行

例如：

- 本地知识库
- 终端设备
- 内网助手
- 工业设备
- 边缘 AI

### 场景 5：需要避免供应商锁定

如果你的公司希望保留模型层的自主权，开源模型更适合作为长期能力建设。

---

## 13. 混合策略：不要二选一

现实项目里，不一定非要开源或闭源二选一。

更常见的是混合架构。

例如：

```text
简单任务 → 小开源模型
复杂任务 → 闭源强模型
敏感数据 → 私有化模型
通用写作 → API 模型
高频 FAQ → 缓存或小模型
高风险决策 → AI + 人工审核
```

这叫模型路由。

简单理解：

> 不同任务，分配给不同模型。

这样可以在质量、成本和安全之间取得平衡。

---

## 14. 什么是模型路由？

模型路由就是：

> 系统先判断任务类型，再选择合适的模型处理。

例如 AI 客服：

| 用户问题 | 路由到哪里 |
|---|---|
| “怎么改密码？” | FAQ / 小模型 |
| “我的订单为什么退款失败？” | RAG + 工具调用 |
| “我要投诉并要求赔偿” | 大模型 + 人工兜底 |
| “总结这份合同风险” | 强模型 + 引用来源 |
| “查询物流状态” | 工具调用，不靠模型猜 |

模型路由是 AI 产品降本增效的重要方法。

---

## 15. 选型时看哪些指标？

选模型时，不要只看排行榜。

产品经理应该看这些维度：

### 质量

模型是否能完成你的真实任务？

### 稳定性

同样输入是否能稳定输出可用结果？

### 延迟

用户等多久能拿到回答？

### 成本

每次请求多少钱？大规模后多少钱？

### 上下文长度

能不能处理你的文档、聊天记录或代码长度？

### 工具调用能力

能不能稳定调用函数或外部系统？

### 部署方式

是否需要云端、私有化、本地或边缘？

### 合规

数据能不能发出去？日志怎么存？供应商协议是否合规？

### 许可证

开源模型能不能商用？有没有限制？

---

## 16. 不要只看 benchmark

Benchmark 是模型评测榜单或测试集。

它有参考价值，但不能完全代表你的产品场景。

一个模型在榜单上高，不代表它一定适合你的业务。

你真正需要的是：

> 用自己的真实任务做评测。

例如你做 AI 客服，就应该准备：

- 真实用户问题
- 正确答案
- 不同难度样本
- 高频问题
- 边界问题
- 投诉问题
- 需要查订单的问题

然后比较不同模型的效果。

---

## 17. 一个 AI 客服选型例子

假设你要做电商 AI 客服。

需求包括：

- 回答退款政策
- 查询订单状态
- 处理投诉
- 总结客服对话
- 生成工单
- 控制成本

可能方案是：

```text
退款政策 → RAG + 中等模型
订单状态 → 工具调用
简单 FAQ → 小模型或缓存
复杂投诉 → 强模型 + 人工审核
客服对话总结 → 闭源强模型或微调开源模型
内部质检 → 批量推理 + 低成本模型
```

这比“所有问题都丢给一个最大模型”更合理。

---

## 18. 开源模型不等于免费

这是一个常见误区。

开源模型虽然可能不用按 token 付 API 费，但你仍然要付：

- GPU 成本
- 服务器成本
- 运维成本
- 工程人力
- 监控系统
- 模型评估
- 安全治理

所以真正要算的是：

> 总拥有成本，Total Cost of Ownership，简称 TCO。

简单说：

> 不是买菜多少钱，而是开餐厅总共多少钱。

---

## 19. 闭源模型不等于不可控

闭源模型也不是完全不能用于严肃业务。

很多服务商会提供：

- 企业版
- 数据不用于训练的选项
- 日志保留设置
- 私有网络连接
- 合规认证
- 访问控制

但产品经理不能默认安全。

你需要和安全、法务、工程一起确认。

---

## 20. 本课总结

闭源模型像点外卖：快、方便、质量通常强，但可控性和长期成本需要评估。

开源模型像自建厨房：更可控、更适合私有化和深度定制，但需要工程和运维能力。

产品早期通常适合用闭源模型快速验证。

当你遇到数据合规、成本、私有化、深度微调或供应商锁定问题时，可以考虑开源模型。

现实中最常见的答案不是二选一，而是：

> 混合使用，按任务路由。

---

## 21. 心智模型

把闭源模型想成：

> 点外卖。

把开源模型想成：

> 自建厨房。

把模型路由想成：

> 不同菜交给不同厨师做。

简单菜交给快厨师，复杂菜交给大厨，敏感菜在自家厨房做。

---

## 22. 初学者常见错误

### 错误 1：以为开源模型就是免费

不是。

你还要付 GPU、服务器、运维和工程成本。

### 错误 2：以为闭源模型一定不能商用

不一定。

很多闭源模型可以商用，但要看服务条款、数据政策和合规要求。

### 错误 3：只看模型排行榜

排行榜只是参考。

真实产品要用自己的数据评测。

### 错误 4：所有任务都用同一个模型

不一定。

简单任务用小模型，复杂任务用强模型，高风险任务加人工审核，通常更合理。

### 错误 5：忽略许可证

开放权重不等于可以随便商用。

一定要看模型 license。

---

## 23. 小练习

判断下面场景更适合闭源模型、开源模型，还是混合方案：

```text
1. 两周内做一个 AI 写作 MVP。
2. 银行内部知识库，数据不能出内网。
3. 每天 100 万次简单客服 FAQ。
4. 复杂法律合同审查，要求高质量并带引用。
5. 手机端离线语音助手。
```

参考答案：

```text
1 → 闭源模型
2 → 开源模型 / 私有化部署
3 → 开源小模型 / 缓存 / 混合方案
4 → 强模型 + RAG + 人工审核，可能是混合方案
5 → 小型开源模型 / 端侧模型
```

## 24. 小项目

为一个 AI 产品做模型选型表。

场景：AI 产品经理知识库助手。

| 任务 | 推荐模型策略 | 原因 |
|---|---|---|
| 总结公开文章 | 闭源模型 | 上手快，质量高 |
| 查询内部文档 | 私有 RAG + 可控模型 | 注意数据安全 |
| 高频简单问答 | 小模型或缓存 | 控制成本 |
| 复杂方案评审 | 强模型 | 需要更好推理和表达 |
| 离线演示版本 | 开源本地模型 | 不依赖网络 |

练习目标：

> 不要问“哪个模型最好”，而是问“哪个模型最适合这个任务”。
